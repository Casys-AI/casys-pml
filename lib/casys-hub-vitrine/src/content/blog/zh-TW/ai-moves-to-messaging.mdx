## 一個沒人真正需要的介面

關於 ChatGPT、Claude 以及所有專用 AI 聊天應用，有一個令人不安的事實：它們都是過渡性介面。

想想看，我們已經有了為對話而生的應用 — WhatsApp、Telegram、LINE、微信、KakaoTalk。數十億人每天開啟它們幾十次。然而，AI 行業卻認為，和 AI 對話需要一個*全新的應用*、一個新賬號、一個新習慣。

對早期使用者來說，這行得通。對其他所有人來說，行不通。

## 觸目驚心的使用者差距

先看資料。

| 平臺 | 日活躍使用者 |
| ---- | ---------- |
| WhatsApp | 23 億 |
| ChatGPT | 1.22 億 |

**WhatsApp 的日活使用者是 ChatGPT 的 27 倍。**不是月活 — 是*日活*。

即便把市面上所有專用 AI 聊天應用加在一起 — ChatGPT、Claude、Gemini、Perplexity，全部算上 — 仍然比即時通訊平臺的使用者規模低一個數量級。

[對話式 AI 市場](https://springsapps.com/knowledge/conversational-ai-trends-in-2025-2026-and-beyond/)預計到 2030 年將達到 413.9 億美元。但市場不會主動走向一個新應用。AI 必須去到人們已經在的地方。

## KakaoTalk 先行一步

如果你覺得"AI 進入即時通訊"還只是空談，那說明你沒有關注韓國的動態。

[KakaoTalk](https://www.kakaocorp.com/page/detail/11865?lang=ENG)，幾乎每個韓國人都在使用的聊天軟體（5300 萬月活），推出了 **PlayMCP** — 即時通訊平臺中首個 MCP 原生整合。它能做什麼：

- **Toolbox（工具箱）**：使用者從集中化的工具市場選擇和管理 MCP 工具，只需一個 Kakao 賬號即可訪問
- **跨平臺**：PlayMCP 工具相容 ChatGPT、Claude 等多個 AI 服務 — 不僅限於 Kakao 自家模型
- **Kanana**：Kakao 的端側 AI 助手，能理解聊天上下文並主動建議操作
- **Kakao Agent**：將即時通訊、地圖、預訂和音樂（Melon）整合到統一的 AI 層

應用場景既日常又強大："把我們剛討論的內容發到我的備忘錄裡。""今天的日程是什麼？""播放去年這個時候我在 Melon 上聽的歌。"

這不是演示。這是一個擁有數千萬使用者的即時通訊平臺，在生產環境中交付 MCP 原生 AI 工具。KakaoTalk 是這種模式可行的[活證明](https://www.kakaocorp.com/page/detail/11725?lang=ENG)。

## WhatsApp：政治性封鎖

你可能以為，擁有 33 億月活使用者的 WhatsApp 是 AI 智慧體的天然歸宿。Meta 另有打算。

2026 年 1 月，Meta [封禁了第三方通用 AI 聊天機器人](https://techcrunch.com/2025/10/18/whatssapp-changes-its-terms-to-bar-general-purpose-chatbots-from-its-platform/)在 WhatsApp Business API 上的接入。OpenAI、Perplexity、Luzia — 全部被切斷。WhatsApp 裡唯一被允許的通用 AI 助手？Meta AI。

各方反應迅速：
- **歐盟**啟動了[反壟斷調查](https://ec.europa.eu/commission/presscorner/detail/en/ip_25_2896) — 最高可處以全球營收 10% 的罰款
- **義大利**[要求 Meta 暫停](https://techcrunch.com/2025/12/24/italy-tells-meta-to-suspend-its-policy-that-bans-rival-ai-chatbots-from-whatsapp/)該政策
- **巴西**也發出了[類似的暫停令](https://techcrunch.com/2026/01/13/brazil-orders-meta-to-suspend-policy-banning-third-party-ai-chatbots-from-whatsapp/)

行業聊天機器人 — FAQ、訂單追蹤、客戶支援 — 仍然被允許，即使它們底層使用了 AI。但基於 MCP 的通用 AI 智慧體？被禁止。這條邊界被刻意模糊化，本質上是政治驅動的。

社群的回應來得很快。[OpenClaw](https://github.com/AidenYangX/OpenClaw)——一個自託管的橋接工具，將 Claude 連線到 WhatsApp、Telegram 和 Signal——在三個月內獲得了 13.5 萬 GitHub 星標。沒有富介面，沒有小程式——只有聊天。但需求訊號不容忽視。

WhatsApp 是地球上最大的即時通訊平臺，卻是對開放 AI 整合最不友好的。這個矛盾將定義未來兩年的行業走向。

## Telegram：開放前沿

WhatsApp 在封鎖，Telegram 在開放。

Telegram 擁有 10 億月活和 5 億日活，其生態系統天然適合這個時刻：

- 對 AI 機器人**沒有任何限制** — ChatGPT、Claude 以及數十種 AI 機器人已經在自由執行
- **Telegram Mini Apps（TMA）**：在聊天介面內執行的完整 Web 應用，使用沙箱化 iframe
- **支付**：原生支援 Google Pay、Apple Pay、加密貨幣
- **Bot + Mini App 組合**：Bot 負責通知和自動化，Mini App 提供豐富的使用者介面

關鍵洞察在這裡：Telegram Mini Apps 與 [MCP Apps](https://blog.modelcontextprotocol.io/posts/2026-01-26-mcp-apps/) 在*技術上完全一致*。兩者都是在沙箱化 iframe 中渲染的 Web 應用，透過訊息傳遞進行通訊。架構完全相同。MCP 到 TMA 的介面卡不是什麼登月工程 — 它只是連線兩個已經共享相同基礎的系統的一座橋樑。

沒有政治封鎖。成熟的 Bot 生態。5 億使用者已經在使用 Mini App。Telegram 是整合 MCP Apps 的理想平臺。

## 超級應用的先例

亞洲多年前就想明白了這件事。

**微信**擁有 14 億月活使用者和 430 萬個小程式。每天有 4.5 億人使用微信小程式，人均每天使用 9.8 個小程式。騰訊正在將[智慧體 AI](https://aiproem.substack.com/p/ai-enhancing-wechats-super-app-status) 直接嵌入微信 — AI 跨服務自動執行任務，對使用者完全透明。AI 不會宣告自己的存在，它只是讓一切運轉得更好。

**LINE** 在日本（9900 萬使用者，佔總人口 78.7%）、泰國和臺灣佔據主導地位。[LINE Mini Apps](https://omisoft.net/blog/next-mini-app-revolution-line-mini-app-development-and-how-to-conquer-new-markets-before-competitors-do/) 是直接從聊天介面啟動的輕量級 Web 應用，支援 LINE Pay 支付。LINE AI 助手 — 基於訂閱制，由 OpenAI 驅動 — 已經在聊天體驗中提供 AI 能力。

這些平臺證明了一件事：人們不想離開聊天軟體去做事。AI 只是下一個他們不應該為之離開的東西。

## MCP Apps：缺失的那座橋

[MCP Apps](https://modelcontextprotocol.io/docs/extensions/apps) 是 Model Context Protocol 的首個官方擴充套件，允許 MCP 工具返回**互動式使用者介面** — 直接在對話中渲染，而不僅僅是純文字。

架構如下：
- 工具宣告 `_meta.ui.resourceUri`，指向一個 UI 資源
- 資源透過 `ui://` 協議以 HTML/JS bundle 的形式提供
- 通訊透過 JSON-RPC over postMessage 在沙箱化 iframe 和宿主之間進行

今天，MCP Apps 已在 Claude、ChatGPT、VS Code、Goose 和 Postman 中執行。但關鍵在於 — 這套架構可以直接對映到即時通訊平臺的 Mini App：

<ExcalidrawDiagram src="mcp-client-bridge" alt="MCP Client Bridge 架構" />

缺失的那塊拼圖是 **MCP Client Bridge** — 一個將 Mini App 互動轉譯為標準 MCP 協議呼叫的介面卡。對於 KakaoTalk，這座橋已經存在（PlayMCP）。對於 Telegram 和 LINE，還需要構建。

但真正難的不是這座橋。真正難的是設計協議 — MCP 已經做到了。真正難的是證明這種模式可行 — KakaoTalk 已經做到了。剩下的，只是工程對接。

## 對開發者意味著什麼

如果你今天在構建 MCP 伺服器，你正在為比你想像的更多的場景構建。

一個 MCP 伺服器可以服務於：
- Claude 和 ChatGPT（透過 MCP Apps）
- VS Code 和 JetBrains（透過 IDE 擴充套件）
- KakaoTalk（透過 PlayMCP，已可用）
- Telegram（透過 Mini App 橋接，即將到來）
- LINE（透過 LIFF SDK 橋接，即將到來）

構建一次能力，部署到所有人們已經在對話的地方。

這正是平臺專用 API 從未兌現的承諾。Slack 的整合在 Teams 裡不能用。WhatsApp 的 Bot 在 Telegram 裡不能用。但 MCP 伺服器呢？它說的是通用協議。適配的是客戶端，不是伺服器。

## 預測

未來 12 到 18 個月內：

- **Telegram** 將為其 Mini Apps 提供官方或社群驅動的 MCP 支援
- **LINE** 將把 MCP 整合到其 CLOVA/AI Assistant 生態中
- **WhatsApp** 將繼續封鎖，除非歐盟和巴西的監管機構強制要求開放 — 而他們很可能會這麼做
- **微信**將開發自己的等效標準（不是直接採用 MCP，但具備互操作性）
- **MCP Apps** 將成為富互動 AI 介面的標準，無論宿主客戶端是什麼

專用 AI 聊天應用不會消失。但它會變成移動網際網路之後的桌面瀏覽器：依然有用，但不再是預設選擇。

預設選擇將是你已經和人聊天的地方。那就是你的即時通訊軟體。

## 真正的戰場

技術問題基本已經解決。MCP 存在了。MCP Apps 存在了。Mini App 平臺存在了。架構對映清晰。

真正的戰場是政治層面的。Meta 想要擁有 WhatsApp 中的 AI 層。Apple 會想要擁有 iMessage 中的 AI 層。Google 會把 Gemini 推進 Google Messages。

但那些保持開放的平臺 — Telegram、KakaoTalk、LINE — 將吸引那些想要觸達數十億使用者、而不必向守門人請求許可的開發者。

摩擦力的論證簡單而終極：摩擦力最小的 AI 介面勝出。而最小的摩擦力，就是*根本不需要新應用*。
