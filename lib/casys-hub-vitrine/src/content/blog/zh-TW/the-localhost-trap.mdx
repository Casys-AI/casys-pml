# Localhost 陷阱

> 你部署了代理。你部署了編排器。它們在同一台機器上。它們無法通訊。歡迎來到生產環境。

---

## 你為什麼在這裡

在[上一篇文章](/blog/process-is-the-prompt)中，我們論證了 AI 代理需要流程——真正的效能提升不是來自更好的模型，而是來自結構化的工作流。我們展示了 Windmill、n8n、Kestra 和 Temporal 等工具如何給代理提供它們一直缺少的東西：排序、錯誤處理、重試、人工審批關卡。海龜圖應用於大語言模型。

很好。於是你照做了。你把代理接入了編排器。你構建了工作流。你覺得自己很聰明。

然後你嘗試部署。一切都崩了。

不是工作流的問題。不是代理的問題。不是模型的問題。**是網路。** 最基礎的那一層——你從來不會想到的那一層——結果成了「在我筆記本上能跑」和「在生產環境能跑」之間的唯一障礙。

這篇文章是我用 Docker 編排器將 OpenClaw 部署到生產環境時學到的經驗。它適用於 Windmill、n8n、Kestra、Temporal、Prefect——任何執行在容器中且需要與主機上的服務通訊的東西。它也適用於 MCP 伺服器、Ollama、本地資料庫，以及基本上任何繫結到 `localhost` 的服務。

如果你正在部署生產環境的 AI 代理，你**一定會**撞到這堵牆。這是地圖。

---

## 起因：為什麼一切都進了 Docker

讓我給你講講你是怎麼走到這一步的，因為路徑總是一樣的。

**第一步：** 你在 VPS 上安裝 OpenClaw。閘道器執行在主機上，監聽 `localhost:18789`。代理連接到 WhatsApp、Telegram 等。一切正常。

**第二步：** 你意識到你的代理需要結構化的工作流——不只是聊天，而是帶有錯誤處理、排程和人工審批的多步驟流程。你選了一個編排器：Windmill、n8n、Kestra、Temporal。它們都以 Docker 映像的形式發布。這就是它們被設計的部署方式。

**第三步：** 你的編排器工作流需要呼叫 OpenClaw 閘道器——觸發代理、發送訊息、檢查狀態。你配置了 webhook URL：`http://localhost:18789`。

**第四步：** 不工作。工作流在 30 秒後逾時。沒有錯誤。沒有 "connection refused"。只有沉默。

你花了兩個小時除錯應用程式，才意識到：**網路才是問題所在。**

## 到底發生了什麼

<ExcalidrawDiagram src="localhost-trap" alt="Localhost 陷阱：127.0.0.1 表示『本機』——但每個容器都是自己的機器" />

`127.0.0.1` 是計算機領域最超載的位址。它表示「本機」——除了它不表示的時候。

當 Docker 建立一個容器時，它建立了一個**網路命名空間**。容器獲得自己的迴環介面，自己的 `127.0.0.1`。當你 Docker 內的編排器呼叫 `localhost` 時，它是在和*自己*說話——不是和主機。不是和你的閘道器。和任何有用的東西都無關。

這不是 bug。這是設計如此。網路隔離是容器的一半意義所在。但它製造了一個看不見的陷阱，因為同一個位址根據你所處的位置意味著不同的東西。

而這就是為什麼它對 AI 基礎設施特別重要：**整個代理技術棧都假設本地通訊。**

- OpenClaw 的閘道器繫結到 `localhost`。
- MCP 伺服器繫結到 `localhost`。
- Ollama 繫結到 `localhost`。
- 工具伺服器、嵌入 API、本地向量儲存——全在 `localhost` 上。

[MCP 規範](https://modelcontextprotocol.io)描述了 `stdio` 和 `SSE` 傳輸方式——`stdio` 按定義需要同一主機，雖然 SSE *技術上*可以透過網路工作，但實際上大多數 MCP 伺服器監聽在 `127.0.0.1`。當你容器化這個技術棧的*任何*部分時，這個假設就崩潰了。

## 四個生產場景

以下是你在實踐中會遇到的所有組合，從最簡單到最複雜。

### 場景 1：主機閘道器 + Docker 編排器

最常見的情況。OpenClaw 直接執行在主機上。Windmill / n8n / Kestra / Temporal 執行在 Docker 中。

<ExcalidrawDiagram src="scenario-1-host-orchestrator" alt="場景 1：主機閘道器透過 socat 橋接轉發到 Docker 編排器" />

**問題：** 容器無法存取主機上的 `localhost:18789`。
**解決方案：** 在 Docker 網路的閘道器 IP 上放一個 socat 橋接。

### 場景 2：主機閘道器 + Docker 代理沙箱

你在隔離的 Docker 容器中執行客戶代理——每個客戶有自己的沙箱，有自己的檔案系統、工具和環境。代理需要回呼 OpenClaw 閘道器。

<ExcalidrawDiagram src="scenario-2-host-sandboxes" alt="場景 2：主機閘道器透過 socat 橋接轉發到多個 Docker 代理沙箱" />

**問題：** 與場景 1 相同，但有多個容器——可能在不同的 Docker 網路上。
**解決方案：** 需要存取的**每個 Docker 網路**一個 socat 橋接。如果所有沙箱共享一個網路，一個橋接就夠了。

### 場景 3：Docker 閘道器 + Docker 編排器

OpenClaw 和編排器都在 Docker 中執行，但在**不同的 Docker 網路**上。當你透過不同的 Compose 檔案分別安裝它們時就會發生這種情況。

<ExcalidrawDiagram src="scenario-3-docker-docker" alt="場景 3：兩個 Docker 容器在共享網路上，直接 DNS 解析" />

**問題：** 不同 Docker 網路上的容器看不到彼此。
**解決方案：** 建立一個共享的 Docker 網路並將兩個服務都連接上去。不需要 socat——只需 Docker 到 Docker 的 DNS 解析。或者如果你不想修改 Compose 檔案，也可以用 socat。

```bash
docker network create shared_internal
# 添加到兩個 compose 檔案：
# networks:
#   shared_internal:
#     external: true
```

### 場景 4：多租戶 — docker0 上隔離的客戶實例

真正的生產用例。每個客戶在獨立的 Docker 容器中擁有自己的 OpenClaw 實例。所有容器執行在預設的 `bridge` 網路（docker0）上。它們透過共享的 socat 橋接存取主機服務（主閘道器、Windmill、Ollama……）。它們**看不到彼此** —— 這是 docker0 的預設隔離。

<ExcalidrawDiagram src="scenario-4-multitenant" alt="場景 4：多租戶架構，客戶容器在 docker0 上隔離，共享 socat 橋接到主機服務" />

**為什麼不用共享的 Docker 網路？** 如果你把所有客戶實例放在同一個自訂 Docker 網路上，它們可以透過容器名互相解析。客戶 A 可以存取客戶 B 的 API、讀取其環境變數，甚至更糟。在多租戶環境中，共享網路是一個**安全漏洞**。

**為什麼用 docker0 + socat 橋接？**

1. **客戶間隔離。** docker0（預設橋接）不提供容器間的 DNS 解析。容器無法發現或存取彼此——這正是多租戶所需要的。
2. **共享橋接。** 每個埠一個 socat 橋接服務*所有*客戶。不需要每個客戶一個橋接——橋接監聽在 docker0 的閘道器 IP（172.17.0.1）上，docker0 上的每個容器都能存取。
3. **無需多客戶 Compose。** 每個客戶是一個獨立的 `docker run`。沒有共享的 Compose 檔案，沒有網路糾纏，沒有「重啟一個客戶導致另一個崩潰」的場景。

```bash
# 啟動客戶實例——預設落在 docker0 上
docker run -d --name client-acme openclaw/sandbox

# 一個 socat 橋接服務 docker0 上的所有客戶
# socat-bridge-docker0-18789 → 轉發到主機閘道器
# socat-bridge-docker0-8000  → 轉發到主機 Windmill
```

客戶共享基礎設施（閘道器、編排器、LLM），但看不到彼此的資料或流量。隔離而不重複。

## 顯而易見的解決方案（以及為什麼它們是錯的）

在找到正確的方法之前，我把別的方法都試了。省省你的時間吧。

**「繫結到 0.0.0.0。」**

這會把你的閘道器暴露給整個網路。如果你在 VPS 上，那就意味著暴露到網際網路。你的閘道器——控制你 AI 代理的東西，能存取你的工具、訊息帳戶、檔案——現在任何掃描你連接埠的人都能存取到。

你可以新增防火牆規則。你會忘掉某一條。或者 Docker 會[好心地繞過 iptables](https://docs.docker.com/engine/network/packet-filtering-firewalls/)，插入自己的規則。

**「用 Docker `host` 網路模式。」**

`network_mode: host` 完全移除容器的網路隔離。容器共享主機的網路堆疊。你的編排器現在可以存取 `localhost:18789` 了——但它也能存取主機上的所有其他服務。你用一個問題換了一個更大的問題。而且你破壞了自己服務間的 Docker Compose 網路。

**「用 `host.docker.internal`。」**

Mac 和 Windows 上的 Docker Desktop 提供這個 DNS 名稱。Linux 不提供——至少不是預設的。你可以在 Compose 檔案中新增 `extra_hosts: ["host.docker.internal:host-gateway"]`，但這透過 Docker 橋接路由，你的主機服務仍然需要在那個橋接的 IP 上監聽。你移動了問題，沒有解決它。

## 真正的問題

真正的問題是架構性的。你有：

1. 一個繫結到 `127.0.0.1` 的**主機服務**（你的 AI 閘道器）
2. 一個需要存取它的**容器化服務**（你的編排器）
3. 主機服務**不能暴露**到網路的要求

這三個約束在標準 Docker 網路下是互斥的。你可以滿足其中任意兩個，但不能同時滿足三個。

除非你加一個橋接。

---

## Socat：無聊但有效的解決方案

[socat](http://www.dest-unreach.org/socat/) 是一個 2004 年的 Unix 工具。它連接兩個資料流。就這樣。它不智慧。它不花俏。它就是一根兩端有位址的管道。

以下是它如何解決我們的問題：

<ExcalidrawDiagram src="socat-bridge" alt="Socat 橋接：容器 → Docker 橋接閘道器 (172.19.0.1) → socat → 主機迴環 (127.0.0.1)" />

一行程式碼：

```bash
socat TCP-LISTEN:18789,bind=172.19.0.1,fork,reuseaddr TCP:127.0.0.1:18789
```

socat 監聽 Docker 橋接 IP——Docker 網路的閘道器位址——並將每個連線轉發到主機的迴環介面。容器可以存取 `172.19.0.1`，因為那是它的預設閘道器。主機服務保持在 `127.0.0.1`。沒有任何東西暴露到外部。

**三個約束同時滿足。** 主機服務保持在 localhost。容器可以存取它。網路看不到任何東西。

## 完整配置 — 自文件化命名

沒有防火牆規則的 socat 不工作——封包會被靜默丟棄。沒有命名規範，三個月後你會面對 `socat-bridge-1`、`socat-bridge-2`，完全不知道哪個是哪個。

以下是我們使用的生產配置：**每個（網路，連接埠）一個橋接**，命名為 `socat-<來源網路>-<目標服務>-<連接埠>`，三個月後你還能看懂每個橋接的作用。

<ExcalidrawDiagram src="socat-bridge-itil" alt="自文件化 socat 橋接：每個（網路，連接埠）一個，所有容器共享" />

### 命名規範

```
socat-bridge-<網路>-<連接埠>
```

示例：
- `socat-bridge-windmill_default-18789` — Windmill workers → OpenClaw 閘道器
- `socat-bridge-sandboxes-18789` — 客戶沙箱 → OpenClaw 閘道器
- `socat-bridge-windmill_default-11434` — Windmill workers → Ollama

一個橋接，該網路上的所有容器共享。如果一個容器在多個網路上，它使用透過哪個網路存取服務就用哪個橋接。

### 1. 找到 Docker 網路的閘道器 IP

```bash
# 列出 Docker 網路及其閘道器 IP
docker network inspect windmill_default --format '{{(index .IPAM.Config 0).Gateway}}'
# → 172.19.0.1
```

### 2. 建立 systemd 服務

```ini
# /etc/systemd/system/socat-bridge-windmill_default-18789.service
[Unit]
Description=Socat bridge: windmill_default:18789 -> localhost:18789 (OpenClaw Gateway)
After=network.target docker.service

[Service]
Type=simple
ExecStart=/usr/bin/socat TCP-LISTEN:18789,bind=172.19.0.1,fork,reuseaddr TCP:127.0.0.1:18789
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
systemctl daemon-reload
systemctl enable --now socat-bridge-windmill_default-18789
```

### 3. 防火牆規則（必需）

沒有這個，socat 在監聽但來自容器的封包會被主機防火牆靜默丟棄。這就是導致 30 秒逾時但沒有錯誤訊息的步驟。

```bash
# 找到 Docker 網路的 Linux 橋接介面
BRIDGE_ID=$(docker network inspect windmill_default --format '{{.Id}}' | cut -c1-12)

# 僅在該橋接介面上允許流量——不暴露到外部
ufw allow in on br-${BRIDGE_ID} to any port 18789 proto tcp comment "socat-bridge-windmill_default-18789"
```

### 4. 測試

```bash
# 從主機——應該已經能工作
curl -s http://127.0.0.1:18789/

# 從該網路上的容器內部
docker run --rm --network windmill_default curlimages/curl \
  curl -s http://172.19.0.1:18789/
```

### 5. 用變數，不要寫死 IP

Docker 橋接 IP 在你重建網路時可能會改變。把閘道器 URL 儲存在你編排器的變數系統中：

```bash
# 在 Windmill 中：建立一個變數
# Path: f/openclaw/gateway_url
# Value: http://172.19.0.1:18789

# 在 n8n 中：使用環境變數
# OPENCLAW_GATEWAY_URL=http://172.19.0.1:18789
```

### 6. 健康檢查腳本

```typescript
// Windmill / Deno / Bun — 在任何編排器中都能用
export async function main() {
  const res = await fetch("http://172.19.0.1:18789/", {
    signal: AbortSignal.timeout(5000)
  });
  return { status: res.status, reachable: res.ok };
}
```

> **多網路 worker。** 一個 Docker 容器可以連接到多個網路——例如 `windmill_default` *和* `shared_internal`。每個網路有自己的橋接 IP。你需要為 worker 所在的**每個網路**建立一個 socat 實例和一條防火牆規則。檢查所有網路：`docker inspect <container> --format '{{json .NetworkSettings.Networks}}'`。

## MCP 伺服器有同樣的問題

如果你在主機上執行 MCP 伺服器並從容器化的代理呼叫它們，這是完全一樣的問題。監聽在 `localhost:3000` 的 MCP 伺服器對任何容器都是不可見的。

socat 橋接模式的工作方式完全相同：

```bash
# MCP 伺服器的橋接
socat TCP-LISTEN:3000,bind=172.19.0.1,fork,reuseaddr TCP:127.0.0.1:3000
```

事實上，**任何**容器需要存取的主機繫結服務都遵循同樣的模式：

| 主機服務 | 容器化用戶端 | 需要橋接 |
| --- | --- | --- |
| OpenClaw 閘道器 | Windmill / n8n / Kestra / Temporal | ✅ |
| MCP 伺服器 | Docker 化的代理沙箱 | ✅ |
| Ollama（本地 LLM） | 任何容器化應用 | ✅ |
| 本地資料庫 | 容器化 API | ✅ |
| 嵌入 API | Docker 中的 RAG 管道 | ✅ |

共同點：一個主機繫結的服務，容器需要存取但不能暴露到網路。一個模式。同樣的解決方案。每次都是。

## 為什麼沒人談這個

AI 基礎設施的討論被協議、模型和框架所主導。MCP、A2A、function calling、tool routing——那些令人興奮的東西。網路是管道工程。

但管道工程讓建築變得宜居。

當我們談論[代理的流程管理](/blog/process-is-the-prompt)時，核心洞察是代理需要編排器。當 Anthropic 發表了他們的[多 Claude 實驗](https://www.anthropic.com/engineering/building-c-compiler)時，協調機制是 Git。當我們討論[代理路由](/blog/routing-agents-without-new-protocol)時，瓶頸是決策層。

所有這些都假設服務之間能夠互相通訊。**沒人提到當它們不能通訊時會發生什麼。** 這是第一步之前的那一步——如此基礎以至於看不見。

直到它崩潰。然後它成了唯一重要的事情。

---

## 令人不安的結論

你的 AI 代理基礎設施中最重要的層不是模型、協議或框架。**是網路。** 具體來說，是 `localhost` 意味著你以為它意味著的那個假設。

每個容器化部署都繼承了這個假設。大多數都會靜默失敗——編排器在執行，閘道器在執行，但回呼永遠不到達，通知永遠發不出，webhook 永遠不觸發。你花幾個小時除錯應用層，才意識到封包根本沒有離開容器。

修復方法是一個 20 年歷史的 Unix 工具、一個 systemd 服務和一條防火牆規則。總配置時間：五分鐘。總配置行數：十二行。

**最好的基礎設施是你配置一次就再也不用想的那種。** Docker 橋接上的 socat 正是如此——無聊、不可見、承重。

就像管道工程應該的樣子。
