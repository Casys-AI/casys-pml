## 一个没人真正需要的界面

关于 ChatGPT、Claude 以及所有专用 AI 聊天应用，有一个令人不安的事实：它们都是过渡性界面。

想想看，我们已经有了为对话而生的应用 — WhatsApp、Telegram、LINE、微信、KakaoTalk。数十亿人每天打开它们几十次。然而，AI 行业却认为，和 AI 对话需要一个*全新的应用*、一个新账号、一个新习惯。

对早期用户来说，这行得通。对其他所有人来说，行不通。

## 触目惊心的用户差距

先看数据。

| 平台 | 日活跃用户 |
| ---- | ---------- |
| WhatsApp | 23 亿 |
| ChatGPT | 1.22 亿 |

**WhatsApp 的日活用户是 ChatGPT 的 27 倍。**不是月活 — 是*日活*。

即便把市面上所有专用 AI 聊天应用加在一起 — ChatGPT、Claude、Gemini、Perplexity，全部算上 — 仍然比即时通讯平台的用户规模低一个数量级。

[对话式 AI 市场](https://springsapps.com/knowledge/conversational-ai-trends-in-2025-2026-and-beyond/)预计到 2030 年将达到 413.9 亿美元。但市场不会主动走向一个新应用。AI 必须去到人们已经在的地方。

## KakaoTalk 先行一步

如果你觉得"AI 进入即时通讯"还只是空谈，那说明你没有关注韩国的动态。

[KakaoTalk](https://www.kakaocorp.com/page/detail/11865?lang=ENG)，几乎每个韩国人都在使用的聊天软件（5300 万月活），推出了 **PlayMCP** — 即时通讯平台中首个 MCP 原生集成。它能做什么：

- **Toolbox（工具箱）**：用户从集中化的工具市场选择和管理 MCP 工具，只需一个 Kakao 账号即可访问
- **跨平台**：PlayMCP 工具兼容 ChatGPT、Claude 等多个 AI 服务 — 不仅限于 Kakao 自家模型
- **Kanana**：Kakao 的端侧 AI 助手，能理解聊天上下文并主动建议操作
- **Kakao Agent**：将即时通讯、地图、预订和音乐（Melon）整合到统一的 AI 层

应用场景既日常又强大："把我们刚讨论的内容发到我的备忘录里。""今天的日程是什么？""播放去年这个时候我在 Melon 上听的歌。"

这不是演示。这是一个拥有数千万用户的即时通讯平台，在生产环境中交付 MCP 原生 AI 工具。KakaoTalk 是这种模式可行的[活证明](https://www.kakaocorp.com/page/detail/11725?lang=ENG)。

## WhatsApp：政治性封锁

你可能以为，拥有 33 亿月活用户的 WhatsApp 是 AI 智能体的天然归宿。Meta 另有打算。

2026 年 1 月，Meta [封禁了第三方通用 AI 聊天机器人](https://techcrunch.com/2025/10/18/whatssapp-changes-its-terms-to-bar-general-purpose-chatbots-from-its-platform/)在 WhatsApp Business API 上的接入。OpenAI、Perplexity、Luzia — 全部被切断。WhatsApp 里唯一被允许的通用 AI 助手？Meta AI。

各方反应迅速：
- **欧盟**启动了[反垄断调查](https://ec.europa.eu/commission/presscorner/detail/en/ip_25_2896) — 最高可处以全球营收 10% 的罚款
- **意大利**[要求 Meta 暂停](https://techcrunch.com/2025/12/24/italy-tells-meta-to-suspend-its-policy-that-bans-rival-ai-chatbots-from-whatsapp/)该政策
- **巴西**也发出了[类似的暂停令](https://techcrunch.com/2026/01/13/brazil-orders-meta-to-suspend-policy-banning-third-party-ai-chatbots-from-whatsapp/)

行业聊天机器人 — FAQ、订单追踪、客户支持 — 仍然被允许，即使它们底层使用了 AI。但基于 MCP 的通用 AI 智能体？被禁止。这条边界被刻意模糊化，本质上是政治驱动的。

社区的回应来得很快。[OpenClaw](https://github.com/AidenYangX/OpenClaw)——一个自托管的桥接工具，将 Claude 连接到 WhatsApp、Telegram 和 Signal——在三个月内获得了 13.5 万 GitHub 星标。没有富界面，没有小程序——只有聊天。但需求信号不容忽视。

WhatsApp 是地球上最大的即时通讯平台，却是对开放 AI 集成最不友好的。这个矛盾将定义未来两年的行业走向。

## Telegram：开放前沿

WhatsApp 在封锁，Telegram 在开放。

Telegram 拥有 10 亿月活和 5 亿日活，其生态系统天然适合这个时刻：

- 对 AI 机器人**没有任何限制** — ChatGPT、Claude 以及数十种 AI 机器人已经在自由运行
- **Telegram Mini Apps（TMA）**：在聊天界面内运行的完整 Web 应用，使用沙箱化 iframe
- **支付**：原生支持 Google Pay、Apple Pay、加密货币
- **Bot + Mini App 组合**：Bot 负责通知和自动化，Mini App 提供丰富的用户界面

关键洞察在这里：Telegram Mini Apps 与 [MCP Apps](https://blog.modelcontextprotocol.io/posts/2026-01-26-mcp-apps/) 在*技术上完全一致*。两者都是在沙箱化 iframe 中渲染的 Web 应用，通过消息传递进行通信。架构完全相同。MCP 到 TMA 的适配器不是什么登月工程 — 它只是连接两个已经共享相同基础的系统的一座桥梁。

没有政治封锁。成熟的 Bot 生态。5 亿用户已经在使用 Mini App。Telegram 是集成 MCP Apps 的理想平台。

## 超级应用的先例

亚洲多年前就想明白了这件事。

**微信**拥有 14 亿月活用户和 430 万个小程序。每天有 4.5 亿人使用微信小程序，人均每天使用 9.8 个小程序。腾讯正在将[智能体 AI](https://aiproem.substack.com/p/ai-enhancing-wechats-super-app-status) 直接嵌入微信 — AI 跨服务自动执行任务，对用户完全透明。AI 不会宣告自己的存在，它只是让一切运转得更好。

**LINE** 在日本（9900 万用户，占总人口 78.7%）、泰国和台湾占据主导地位。[LINE Mini Apps](https://omisoft.net/blog/next-mini-app-revolution-line-mini-app-development-and-how-to-conquer-new-markets-before-competitors-do/) 是直接从聊天界面启动的轻量级 Web 应用，支持 LINE Pay 支付。LINE AI 助手 — 基于订阅制，由 OpenAI 驱动 — 已经在聊天体验中提供 AI 能力。

这些平台证明了一件事：人们不想离开聊天软件去做事。AI 只是下一个他们不应该为之离开的东西。

## MCP Apps：缺失的那座桥

[MCP Apps](https://modelcontextprotocol.io/docs/extensions/apps) 是 Model Context Protocol 的首个官方扩展，允许 MCP 工具返回**交互式用户界面** — 直接在对话中渲染，而不仅仅是纯文本。

架构如下：
- 工具声明 `_meta.ui.resourceUri`，指向一个 UI 资源
- 资源通过 `ui://` 协议以 HTML/JS bundle 的形式提供
- 通信通过 JSON-RPC over postMessage 在沙箱化 iframe 和宿主之间进行

今天，MCP Apps 已在 Claude、ChatGPT、VS Code、Goose 和 Postman 中运行。但关键在于 — 这套架构可以直接映射到即时通讯平台的 Mini App：

缺失的那块拼图是 **MCP Client Bridge** — 一个将 Mini App 交互转译为标准 MCP 协议调用的适配器。对于 KakaoTalk，这座桥已经存在（PlayMCP）。对于 Telegram 和 LINE，还需要构建。

但真正难的不是这座桥。真正难的是设计协议 — MCP 已经做到了。真正难的是证明这种模式可行 — KakaoTalk 已经做到了。剩下的，只是工程对接。

## 对开发者意味着什么

如果你今天在构建 MCP 服务器，你正在为比你想象的更多的场景构建。

一个 MCP 服务器可以服务于：
- Claude 和 ChatGPT（通过 MCP Apps）
- VS Code 和 JetBrains（通过 IDE 扩展）
- KakaoTalk（通过 PlayMCP，已可用）
- Telegram（通过 Mini App 桥接，即将到来）
- LINE（通过 LIFF SDK 桥接，即将到来）

构建一次能力，部署到所有人们已经在对话的地方。

这正是平台专用 API 从未兑现的承诺。Slack 的集成在 Teams 里不能用。WhatsApp 的 Bot 在 Telegram 里不能用。但 MCP 服务器呢？它说的是通用协议。适配的是客户端，不是服务器。

## 预测

未来 12 到 18 个月内：

- **Telegram** 将为其 Mini Apps 提供官方或社区驱动的 MCP 支持
- **LINE** 将把 MCP 集成到其 CLOVA/AI Assistant 生态中
- **WhatsApp** 将继续封锁，除非欧盟和巴西的监管机构强制要求开放 — 而他们很可能会这么做
- **微信**将开发自己的等效标准（不是直接采用 MCP，但具备互操作性）
- **MCP Apps** 将成为富交互 AI 界面的标准，无论宿主客户端是什么

专用 AI 聊天应用不会消失。但它会变成移动互联网之后的桌面浏览器：依然有用，但不再是默认选择。

默认选择将是你已经和人聊天的地方。那就是你的即时通讯软件。

## 真正的战场

技术问题基本已经解决。MCP 存在了。MCP Apps 存在了。Mini App 平台存在了。架构映射清晰。

真正的战场是政治层面的。Meta 想要拥有 WhatsApp 中的 AI 层。Apple 会想要拥有 iMessage 中的 AI 层。Google 会把 Gemini 推进 Google Messages。

但那些保持开放的平台 — Telegram、KakaoTalk、LINE — 将吸引那些想要触达数十亿用户、而不必向守门人请求许可的开发者。

摩擦力的论证简单而终极：摩擦力最小的 AI 界面胜出。而最小的摩擦力，就是*根本不需要新应用*。
