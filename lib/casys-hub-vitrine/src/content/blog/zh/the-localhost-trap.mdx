# Localhost 陷阱

> Agent 在跑。編排器在跑。它們在同一台機器上。它們無法通訊。歡迎來到正式環境。

---

**這個場景每週都在上演。** 有人架了一個 AI Gateway — OpenClaw、LiteLLM，隨便哪個 — 跑得好好的。然後加了一個 Docker 裡的編排器 — Windmill、n8n、Temporal — 接上一個需要回呼 Gateway 的工作流。指向 `http://localhost:18789`。

Timeout。三十秒的沉默，然後失敗。

檢查服務。在跑。從主機 curl。立刻回應。從容器裡面 curl。請求掛起，然後死掉。連個乾淨的「connection refused」都沒有 — 只有沉默。封包離開容器，撞上防火牆，然後消失。這就是陷阱：它*看起來*像是服務太慢，而不是網路問題。

服務就在那裡。同一台機器上。在 RAM 裡只隔幾個位元組。但在容器內部，`localhost` 指向的是一台*不同的機器*。

## 那個假設

`127.0.0.1` 是計算機領域中被過度使用的地址。它的意思是「這台機器」— 除了在它不是的時候。

Docker 建立容器時，會建立一個**網路命名空間**。容器有自己的 loopback 介面，自己的 `127.0.0.1`。當 Docker 裡的編排器呼叫 `localhost`，它是在跟*自己*說話 — 不是主機、不是你的 Gateway、什麼都不是。

這不是 bug。是設計。網路隔離是容器的核心價值之一。但它製造了一個直到踩到才看得見的陷阱，因為同一個地址在不同位置代表不同的東西。

而這對 AI Agent 特別重要：**整個 Agent 技術棧都假設本地通訊。**

MCP 伺服器綁定在 localhost。Gateway 綁定在 localhost。Tool Server 綁定在 localhost。[MCP 規範](https://modelcontextprotocol.io)描述的 `stdio` 和 `SSE` 傳輸 — `stdio` 本質上要求同一台主機，而 SSE *技術上*可以跨網路運作，但實際上大多數 MCP 伺服器都監聽在 localhost。只要把這個技術棧的*任何*部分容器化，假設就崩了。

---

## 顯而易見的方案（以及為什麼它們是錯的）

**「綁定到 0.0.0.0。」**

這會把你的 Gateway 暴露給整個網路。如果你在 VPS 上，那就是暴露給整個網際網路。你的 Gateway — 那個控制你 AI Agent、存取你的工具、訊息帳號、檔案的東西 — 任何掃描你 IP 的人都能連上。

**「用 Docker host 模式。」**

`network_mode: host` 完全移除容器的網路隔離。編排器可以連到 `localhost:18789` 了 — 但它也能連到主機上的所有其他服務。你把一個問題換成了更大的問題。

**「用 `host.docker.internal`。」**

Docker Desktop 在 Mac 和 Windows 上提供這個 DNS 名稱。Linux 上沒有 — 至少不是預設的。你可以在 Compose 裡加 `extra_hosts`，但你只是搬移了問題，沒有解決。

## Socat：無聊但有效的解決方案

[socat](http://www.dest-unreach.org/socat/) 是 2004 年的 Unix 工具。它連接兩個資料流。就這樣。

<ExcalidrawDiagram src="socat-bridge" alt="Socat Bridge：容器 → Docker Bridge Gateway (172.19.0.1) → socat → 主機 Loopback (127.0.0.1)" />

一行指令：

```bash
socat TCP-LISTEN:18789,bind=172.19.0.1,fork,reuseaddr TCP:127.0.0.1:18789
```

socat 監聽 Docker Bridge 的 IP — Docker 網路的 Gateway 地址 — 並將每個連線轉發到主機的 loopback。容器能連到 `172.19.0.1` 因為那是它的預設 Gateway。主機服務保持在 `127.0.0.1`。外部什麼都看不到。

**三個限制條件全部滿足。**

但光靠 socat 還不夠。你還需要**防火牆規則** — 沒有的話，封包到達 bridge IP 但會被主機防火牆默默丟棄。這就是讓你浪費好幾個小時的部分：socat 在跑，服務在跑，但容器的請求就是*掛住*。沒有錯誤，沒有拒絕，只有 30 秒的 timeout。解法：一條 UFW 規則限定在 Docker bridge 介面上（見下文）。這不是可選的。

> **注意多網路的 Worker。** 一個 Docker 容器可以連接多個網路 — 例如 `windmill_default` *和* `shared_internal`。每個網路有自己的 bridge IP。你需要為 Worker 連接的**每個網路**都設定 socat 和防火牆規則。檢查所有網路：`docker inspect <container> --format '{{json .NetworkSettings.Networks}}'`。

## 為什麼這對 AI 很重要

這不是 Docker 的問題。這是 **Agent 基礎設施**的問題。

Agent 生態系統正在收斂到一個模式：一個 Gateway 在 Agent 和外部世界之間做中介。OpenClaw 這樣做。LangGraph Cloud 也是。CrewAI、AutoGen 也是。Gateway 持有 Session 狀態、工具策略、訊息通道、記憶體。一切都經過它。

同時，工作流編排器正在成為 Agent 的流程層。我們[之前論證過](/blog/process-is-the-prompt) Agent 需要流程，而編排器提供流程 — Windmill、n8n、Temporal、Prefect。

兩者一起跑，就需要它們互相通訊。而它們做不到 — 因為一個在容器裡，另一個不在。

**每個正式的 AI 部署都會撞上這面牆。** 協議（MCP、A2A）處理應用層。沒人處理網路層 — 那個讓所有東西運作的無聊管道。

## 通用模式

| 主機服務 | 容器化客戶端 | 需要 Bridge |
| --- | --- | --- |
| AI Gateway (OpenClaw, LiteLLM) | 編排器 (Windmill, n8n) | ✅ |
| MCP 伺服器 | Docker 化的 Agent | ✅ |
| 本地 LLM (Ollama) | 任何容器化應用 | ✅ |
| Localhost 資料庫 | 容器化 API | ✅ |

---

## 不舒服的結論

你 AI Agent 基礎設施中最重要的層不是模型、協議或框架。**是網路。** 具體來說，是 `localhost` 代表你以為的意思這個假設。

修復方案：一個 20 年的 Unix 工具、一個 systemd 服務、一條防火牆規則。設定時間：五分鐘。設定行數：十二行。

**最好的基礎設施是裝好就忘掉的那種。** Docker Bridge 上的 socat 正是如此 — 無聊、看不見、但撐住一切。

就像水管該有的樣子。
