# 如果 AI 只是需要一個流程？

> 我在不知不覺中重新發明了流程管理。而且它確實有效。

---

**上週我試用了 OpenClaw。** 這是一個開源橋接工具，能將 Claude 連接到 WhatsApp、Telegram 和 Signal——就是我們在[上一篇文章](/blog/ai-moves-to-messaging)中提到的那個。三個月內獲得了 135,000 個 GitHub 星標。是時候測試了。

三小時後，我已經不在 OpenClaw 裡了。我在 [Windmill](https://www.windmill.dev/)——一個工作流程編排器——裡面為我的代理構建管道。

就在那時，我有了一種似曾相識的感覺。

我正在做的事情，我以前見過。不是在 GitHub 倉庫裡。而是十年前在工程學院的課堂上。

我在給代理提供**文件**。我在分配**工具**。我在組建專業**團隊**。我在衡量他們的**績效**。

不知不覺間，我正在重現品質管理課上學到的東西。

---

## 烏龜圖

有一種工具從 1980 年代起就被工業界用來描述任何流程。它叫做**烏龜圖**——大概是因為從上往下看，它隱約像一隻伸開四條腿的烏龜。它是品質管理的支柱，由 Philip Crosby 在其「零缺陷」理念中推廣開來。

概念很簡單。對於任何流程，問四個問題：

- **用什麼？** ——所需的工具和設備
- **靠誰？** ——技能和角色
- **怎麼做？** ——程序和文件
- **做多少？** ——績效指標

中間是流程本身。左邊是輸入。右邊是輸出。

這是白領們用活頁夾做的東西。工廠的東西。生產線的東西。跟 AI 毫無關係。

然而。

---

## 同樣的形態，無處不在

讓我停下來的是這個。當你把 AI 代理放進工作流程時，你會發現完全一樣的格子。

| 烏龜圖 | 代理編排 |
| --- | --- |
| **用什麼** ——機器、軟體 | MCP tools、APIs、function calling |
| **靠誰** ——受訓人員 | 專業化代理，按任務分配 |
| **怎麼做** ——程序、文件 | System prompts、context files、CLAUDE.md |
| **做多少** ——合格率 | Token 消耗、延遲、幻覺率 |
| **輸入** ——原材料 | 使用者意圖、對話上下文 |
| **輸出** ——合格產品 | 結構化結果、已執行的動作 |

<ExcalidrawDiagram src="turtle-process-agents" alt="The Turtle Diagram: Process Management → AI Agent Orchestration" />

這種對應令人驚訝。不是因為它是牽強的——而是因為它是*自然的*。沒有人坐下來說「讓我們把品質管理套用在 LLM 上」。它自己就發生了。所有認真嘗試過讓代理在工作流程中運作的人，都趨同到了這個模式——甚至不知道它的存在。

同樣的形態在各種情境中獨立湧現。因為這是一個執行指令的實體正確完成工作所需的最小結構——無論它是血肉之軀還是矽基之身。

---

## 言語的經濟學

沒有流程的代理，就是一個以整個潛在空間為遊樂場的 LLM。它可以去任何地方。而這恰恰就是它所做的。它用冗長來彌補框架的缺失——它「打安全牌」，生成更多內容，就像一個沒有職位說明的實習生，什麼都做一點來表示自己有用。

有流程的代理，是一個空間被約束的 LLM。它知道該做什麼、用什麼、如何衡量結果。它產生更少的 token。更相關的 token。

David Byrne 在 1977 年唱道：*"You're talking a lot, but you're not saying anything."* 他說的不是 LLM。但這句歌詞精確地捕捉了這個現象。一個囉嗦的代理不是一個愚蠢的代理。它是一個**沒有流程約束**的代理。

浪費的 token。需要核查的幻覺。第一次輸出離題後的反覆修改。這些不是智慧問題。這些是流程問題。

而流程是免費的。建立流程的成本永遠低於沒有流程的代價。

---

## 編排工具不會說出自己的名字

當我退一步審視 Windmill 讓我做了什麼時，我意識到所有這些工具——Windmill、n8n、Make，以及其他——其實是偽裝成自動化工具的**流程管理工具**。它們所做的就是 BPMN——業務流程模型與標記法——建模業務流程的標準。同樣的方框、同樣的箭頭、同樣的閘道。只不過泳道裡畫的不是人類，而是代理。

看看當你整合一個代理時，它們要求你做什麼：

1. **記錄**上下文和指令
2. **列出**可用的工具和權限
3. **分配**代理到每個步驟
4. **定義**成功和失敗的標準
5. **迭代**當輸出不理想時

這就是品質管理。穿了 YAML 的外衣。

專業化在這裡和其他地方一樣有效：五個專業化代理的管道始終優於一個試圖在單一 prompt 中完成所有工作的通才代理。

## Crosby 的五個層次

Crosby 建立了一個五階段的成熟度矩陣，描述組織如何學會掌控品質——從完全的不確定性到本能反射。當我對照身邊團隊與代理的關係時，我發現了完全相同的層級。

| 成熟度 | 人類流程（Crosby） | 代理編排 |
| --- | --- | --- |
| **1 — 不確定** | 「我們沒有流程」 | 「我們在 ChatGPT 裡複製貼上 prompt」 |
| **2 — 覺醒** | 「我們開始記錄文件」 | 「我們有 system prompt 和一些工具」 |
| **3 — 啟明** | 「我們衡量並改進」 | 「有監控的結構化工作流程」 |
| **4 — 智慧** | 「流程已融入文化」 | 「用於編排的 CI/CD 管道」 |
| **5 — 確信** | 「品質已是本能」 | 「自我評估的代理，自適應工作流程」 |

<ExcalidrawDiagram src="process-maturity-agents" alt="Crosby's Quality Maturity Grid → Agent Orchestration Maturity" />

我認識的大多數人都在第 1 級和第 2 級之間。那些認真用代理構建產品的人在第 3 級。第 5 級，我還沒在任何地方見過。

從經驗中得出的幾點觀察：

**流程過多反而扼殺流程。** 過多的上下文會飽和 context window。效能下降。最優的流程不是最多的流程——而是*足夠的*流程。一種「夠好就好」，不是套用在[個性](/blog/personality-is-just-friction)上，而是套用在結構上。

**過度專業化帶來脆弱性。** 超專業化的代理工作流程在任何一個步驟失敗時就會崩潰。韌性來自專業化與適應性之間的平衡。

**這些都不是新觀點。** 幾十年來都有記載。我們不需要發明新理論來驅動代理。我們需要重讀已知的知識。

## 沒有人問的問題

為什麼沒有人談論這個？

也許是因為「給代理一個流程它們就會表現得更好」是一個極其無聊的結論。它不是能登上 Hacker News 頭版的論文。不是驚人的基準測試。不是有著五個字母縮寫的新模型。

**結構勝過天賦。** 一個一般的模型在好的流程中，勝過一個沒有流程的優秀模型。這對人類是如此。對代理更是如此——因為代理*就是*流程。它們執行指令。它們消費輸入。它們產出結果。烏龜圖對它們來說不是比喻。它是一份**規格書**。

---

## 品質是免費的。Token 不是。

從 Windmill 的探索中回來後，讓我印象深刻的不是代理「需要管理」。而是**做好工作的結構不取決於誰來做**——人類、機器人、AI 代理。它取決於工作本身的性質。

2026 年，翻譯再清楚不過：

- 一個結構良好的工作流程的成本：幾個小時的設計
- 一個沒有流程的代理的成本：數千個浪費的 token、需要核查的幻覺、修正輸出的人工返工

模型在變。框架在變。工具在變。但做好工作的結構，不會變。

品質工程師在 1980 年代就知道了。我們只是用 GPU 重新發現了它。
