# Epic 10 Retrospective - DAG Capability Learning & Unified APIs

**Date**: 2026-02-24
**Scope**: Full retrospective with quantitative analysis (supersedes preliminary retro of 2025-12-26)
**Status**: PARTIAL - 8/9 stories delivered, 10-8 remains in backlog

---

## 1. Epic Summary

### What Was Delivered

Epic 10 unified two separate execution models (explicit DAG and free-form code) into a single coherent learning system. The core deliverables:

- **Static analysis pipeline** (10.1, 10.2) that parses TypeScript via SWC to produce a complete `static_structure` with nodes, edges, arguments, and control flow -- creating Capabilities before execution.
- **Data flow graph** (10.3) with `provides` edge type capturing schema-based tool-to-tool data relationships with coverage semantics (strict/partial/optional).
- **Unified execution architecture** (10.5) routing ALL execution through WorkerBridge with permissions="none" for 100% traceability. Worker was 1.7x faster than subprocess.
- **Two unified APIs** replacing 5 fragmented tools:
  - `pml_discover` (10.6) -- unified search across tools and capabilities.
  - `pml_execute` (10.7) -- unified execution with automatic capability learning, integrating SHGAT multi-head attention, DR-DSP hypergraph pathfinding, and ControlledExecutor.
- **Thompson Sampling** (10.7c) for per-tool Bayesian threshold management with scope-based risk classification and Smart HIL.
- **SHGAT persistence** (10.7b, merged into 10.7) saving/loading attention weights across restarts.

### Story Merges

Three stories were merged into parent stories during implementation:

| Original Story | Merged Into | Rationale |
|---|---|---|
| 10.7a (DR-DSP Integration) | 10.7 | DR-DSP was natural part of pml_execute suggestion mode |
| 10.7b (SHGAT Scoring) | 10.7 | SHGAT scoring was inseparable from suggestion mode |
| 10.6a (Unified Search) | 10.6 | Scoring formula was part of discover handler |

This reduced cognitive overhead and avoided artificial story boundaries, but also made 10.7 a large story (~800 LOC handler + 19 modified files).

## 2. What Went Well

### Effective Story Merging

Merging 10.7a (DR-DSP) and 10.7b (SHGAT) into 10.7 avoided handoff friction between stories that shared the same handler. The three algorithms (SHGAT scoring, DR-DSP pathfinding, CapabilityMatcher) were already implemented as POCs -- the stories were pure integration work. Keeping them in one story let the developer wire all three in a single pass with a coherent test suite.

### Architecture Discovery Under Pressure (10.5)

Story 10.5 was planned as a converter (`staticStructureToDag`) + handler modification. During implementation, it revealed the sandbox bypass bug (H4): `createToolExecutor()` in three handlers called `client.callTool()` directly, skipping Worker sandboxing. The fix -- `createToolExecutorViaWorker()` -- established the Worker-only architecture that made all subsequent stories simpler. The discovery cost 2 extra days but saved far more downstream.

### Aggressive Deprecation

Old APIs (`pml_search_tools`, `pml_search_capabilities`, `pml_execute_dag`, `pml_execute_code`) were deprecated immediately in 10.6 and 10.7 with log warnings. No transition period. This kept the codebase clean and prevented dual-maintenance burden.

### Reuse of SWC Patterns

Stories 10.1 and 10.2 heavily reused patterns from the existing SchemaInferrer (726 LOC) and PermissionInferrer (510 LOC): same SWC version, same AST traversal approach, same member chain extraction. The StaticStructureBuilder (800+ LOC) was large but structurally familiar, reducing review effort.

### Code Review Discipline

Every story had formal code review with HIGH/MEDIUM/LOW issue classification:
- 10.5 had **4 HIGH issues** (H1: broken argument resolution, H2: false positive, H3: missing integration test, H4: sandbox bypass). All were resolved before merge.
- 10.6 had **1 HIGH issue** (missing E2E tests) + 5 MEDIUM + 4 LOW. All 10 items resolved.
- 10.7 had duplication identified and extracted into `shared/tool-definitions.ts`, removing ~180 LOC of duplicated code.

## 3. Challenges and Struggles

### The Iceberg Effect on Story 10.5

Estimated at 2-3 days, delivered in 5. The visible scope was "convert static structure to DAG and execute via ControlledExecutor." The hidden scope was:

1. Discovering the sandbox bypass in 3 separate handlers.
2. Creating `workerbridge-executor.ts` as a new execution path.
3. Unifying `DenoSandboxExecutor.execute()` to use Worker by default (AC13).
4. Running benchmarks (Worker ~31ms vs subprocess ~53ms).
5. Updating 268 existing sandbox tests.

The scope roughly tripled from what was planned. The story grew from 8 ACs to 13.

### 10.7 Became a Mega-Story

After absorbing 10.7a and 10.7b, Story 10.7 touched 19 files, produced an 800 LOC handler, and required coordination of four separate algorithms (SWC analysis, SHGAT multi-head attention, DR-DSP hypergraph pathfinding, ControlledExecutor). This was manageable because the algorithms were already implemented as POCs, but it pushed the limits of a single story's cognitive load.

### Speculation Mode Disabled

A significant design goal of 10.7 -- speculative execution of found capabilities -- had to be disabled. The `static_structure` contains hardcoded arguments from the original execution. Without parameterized capabilities (deferred to Epic 12), the system cannot adapt arguments to new intents. The mode Suggestion returns suggestions instead of executing, which is a meaningful degradation from the original vision.

### Silent Fallback Debate (10.5 M2)

Story 10.5 code review flagged a silent fallback in the DAG execution path: if static analysis fails, it falls back to direct sandbox execution without warning. The team decided this was intentional ("try DAG first, fallback to sandbox for robustness"), but this contradicts the project's No Silent Fallbacks policy. The decision was documented but remains an unresolved tension.

### Feature Creep: AC12 in 10.7

AC12 (SHGAT Live Learning) was added to 10.7 but ultimately moved to Epic 11.6. This is a pattern worth watching: when a large story absorbs sub-stories, there's a tendency to add "one more thing" that doesn't belong.

## 4. Key Lessons Learned

### L1: Architecture discovery stories need explicit scope buffers

When a story touches infrastructure (Worker, RPC, handler routing), add +50% to the estimate. Story 10.5 proved that "modify a handler" can cascade into "redesign the execution architecture." The recommendation from the first retro (buffer +50% for multi-layer stories) is validated by the data.

### L2: POC-first, integrate-second works

DR-DSP (460 LOC), SHGAT (1284 LOC), and Thompson (708 LOC) were all implemented as standalone POC modules before Epic 10. The integration stories (10.7, 10.7c) were fast because they wired existing, tested code. This pattern should be repeated: build algorithms in isolation, test them, then integrate.

### L3: Story merges are fine when the code is in the same handler

Merging 10.7a and 10.7b into 10.7 worked because all three touched `execute-handler.ts`. Merging 10.6a into 10.6 worked because it was the same `discover-handler.ts`. The signal: if two stories will be implemented by the same developer in the same file in the same session, merge them. If they're in different files or different sessions, keep them separate.

### L4: Code review catches real bugs

10.5's H4 (sandbox bypass) was a security-relevant bug. 10.7's duplication fix removed ~180 LOC. These were not cosmetic findings. The pattern of adversarial code review with severity classification (HIGH/MEDIUM/LOW) and explicit resolution is worth preserving.

### L5: Deprecation is cheap, dual maintenance is expensive

Immediately deprecating old tools and removing them from `getMetaTools()` was the right call. The old handlers still work if called directly (backward compat), but they're invisible to new clients. This kept the API surface clean without breaking existing integrations.

## 5. Technical Debt

### 5.1 Story 10.8 (pml_get_task_result) -- Backlog

Still unimplemented. This tool would let an AI agent fetch full task results when the 240-char `resultPreview` is insufficient. Current workaround: the agent must re-execute. Low priority given that most results fit in the preview, but it will matter for large JSON payloads.

### 5.2 Speculation Mode Disabled

`execute-handler.ts:609-621` has a TODO(Epic-12) comment: speculative execution is disabled because capabilities have hardcoded arguments. This means the suggestion mode always returns suggestions, never auto-executes found capabilities. Resolving this requires parameterized capabilities from Epic 12.

### 5.3 Silent Fallback in DAG Execution

The try-DAG-first, fallback-to-sandbox pattern in `code-execution-handler.ts` is intentional but undocumented at runtime. The No Silent Fallbacks policy recommends at minimum a `log.warn()` when fallback occurs. This should be added.

### 5.4 10.5 AC4 Conditional Execution Verification

AC4 (conditional execution support) has a warning note: "Validation missing that `task.condition` is evaluated at runtime." A test creating a conditional DAG and verifying branch skip/include behavior should be added.

### 5.5 SHGAT Feature Creep -> Epic 11.6

AC12 (SHGAT Live Learning) was added to 10.7, then moved to 11.6. The story document still references it. Clean up the story to remove moved-out scope.

## 6. Patterns and Insights

### Static Analysis Pattern Established

Epic 10 established a clear pattern for SWC-based static analysis:

```
Code (TypeScript)
  -> SWC parse (same version across all analyzers)
  -> AST traversal (recursive, discriminated union on node.type)
  -> Domain-specific extraction (tools, arguments, control flow)
  -> Structure generation (nodes + edges)
  -> Integration with CapabilityStore
```

Three analyzers now follow this pattern: SchemaInferrer (Epic 7), PermissionInferrer (Epic 7), StaticStructureBuilder (Epic 10). Any future code analysis (e.g., loop detection, mutation tracking) should extend StaticStructureBuilder rather than creating a new class.

### API Unification Lessons

The 5-to-2 API reduction followed a clear principle: **one tool per intent class**.

| Intent | Old Tools (5) | New Tool (2) |
|---|---|---|
| "Find something" | pml_search_tools, pml_search_capabilities, pml_find_capabilities | `pml_discover` |
| "Do something" | pml_execute_dag, pml_execute_code | `pml_execute` |

The scoring unification (`semantic x reliability` for discover, SHGAT multi-head for execute) was also important: different scoring formulas for different tools confused clients.

### Worker-Only Architecture

The single most impactful discovery of Epic 10: ALL execution routes through WorkerBridge with permissions="none". Benefits measured:

- 100% traceability (every tool call goes through RPC proxy)
- 1.7x performance improvement (Worker ~31ms vs subprocess ~53ms)
- Single code path (no fallback logic to maintain)
- Centralized permission enforcement

### Risk Classification via Scope

Story 10.7c established a mapping from `mcp-permissions.yaml` scope to Thompson risk categories:

| Scope | Risk | Threshold |
|---|---|---|
| minimal, readonly | safe | ~0.55 |
| filesystem, network-api | moderate | ~0.70 |
| mcp-standard | dangerous | ~0.85 |
| unknown | -- | HIL bypass |

This is a reusable pattern for any future risk-based decision making.

## 7. Recommendations for Future Work

### R1: Parameterized Capabilities (Epic 12, Critical)

The suggestion mode returning suggestions instead of auto-executing is the biggest functional gap. Epic 12 should prioritize parameterized capabilities where `arguments` can be adapted to new intents at runtime, enabling true speculative execution.

### R2: Complete 10.8 (pml_get_task_result) if AIL patterns demand it

Monitor whether agents frequently need full results beyond the 240-char preview. If so, implement 10.8. If the preview proves sufficient in practice, close the story as WONTFIX.

### R3: Add log.warn() to DAG fallback path

One-line fix to bring the DAG-to-sandbox fallback into compliance with the No Silent Fallbacks policy. Low effort, high consistency value.

### R4: Validate conditional execution at runtime (10.5 AC4)

Add an integration test that creates a DAG with decision nodes and verifies that the correct branch is taken based on runtime condition evaluation. This was flagged during implementation but never resolved.

### R5: Keep the POC-first pattern for Epic 12 algorithms

If Epic 12 introduces new algorithms (e.g., argument binding, session context resolution), implement them as standalone POC modules first with their own test suites, then integrate. This pattern kept 10.7 integration fast and focused.

### R6: Story size ceiling

10.7 (with merges) touched 19 files and produced ~800 LOC of new handler code. This is near the maximum manageable scope for a single story. If future merges would push past 15 modified files, keep stories separate.
