## What Went Well

### 1. Hybrid Local/Cloud Architecture Delivered End-to-End

The core vision -- one MCP gateway that transparently routes between local sandbox execution and cloud-hosted services -- was realized across 9 stories. The final architecture is clean:

```
Claude Code -> pml stdio -> Permission Check -> Routing -> (Local Sandbox | Cloud HTTP) -> Response
```

Key design wins:
- **Routing is platform-defined** (not user-configurable) via `mcp-routing.json` -- eliminates misconfiguration
- **Approval mode is user-defined** (runtime computed from `.pml.json`) -- respects user autonomy
- **Per-project state** (`${workspace}/.pml/`) instead of global -- isolation between projects

### 2. Security Model: Defense in Depth

Three layered security mechanisms that complement each other:
- **Sandbox isolation** (Deno Worker `permissions: "none"`) -- untrusted capability code cannot access system
- **HIL approval flow** -- user approves dangerous operations via MCP response/callback pattern
- **Lockfile integrity** (`mcp.lock`) -- detects tampered or updated MCP code with hash verification

The decision to sandbox *capability code* (untrusted) while leaving *MCP servers* unsandboxed (they need system access, protected by HIL) was architecturally sound.

### 3. Stateless HIL Pattern

Story 14.3b's stateless approval pattern was a significant simplification over the main codebase's `activeWorkflows` Map approach. Capability metadata contains all context needed for re-execution after approval. No server-side workflow state needed.

**Correction noted**: The initial "stateless" claim was revised -- `PendingWorkflowStore` was ultimately needed for the client side (ADR-062). The honesty in documenting this mid-course correction is good practice.

### 4. Binary Distribution (14-11)

The public repo split (`casys-pml` for CLI + releases, `mcp-std` for MiniTools) solved the private-repo-binary-distribution problem elegantly:
- Secret sauce (GraphRAG, SHGAT) stays private
- Open source CLI and MiniTools (community trust, MIT license)
- GitHub Releases as free CDN for binaries
- `pml upgrade` self-update command

### 5. Test Discipline

731+ tests across the epic with consistent patterns:
- Every story has unit tests for core logic
- Code review follow-ups tracked as issues with severity (HIGH/MEDIUM/LOW) and all resolved
- E2E tests (14-8) cover all 10 acceptance criteria with mock cloud server, stdio simulator, and real sandbox isolation tests

## Challenges & Struggles

### 1. HIL in Stdio Mode Was Non-Trivial

Story 14.4 initially implemented HIL as a blocking `await hilCallback(prompt)` -- which deadlocks because stdin IS the JSON-RPC channel. This required Story 14.3b as a corrective story to implement the `approval_required` response pattern. The blocking-callback approach was a predictable mistake that should have been caught at design time.

**Impact**: 0.5 day corrective story, but the final pattern is cleaner than the original design.

### 2. Scope Creep from Rewrites

Several stories underwent multiple rewrites during implementation:
- **14.6** (BYOK): Started as "HTTP server + BYOK injection", rewritten twice to MVP scope (`.env` only, no cloud profile)
- **14.7** (Registry): Three rewrites -- FQDN format, hash security model, seed JSON vs dynamic derivation

While the final designs are better, each rewrite consumed time. Earlier spikes or design reviews could have reduced churn.

### 3. MiniTool Bundling Deferred (Task 4 of 14-7)

MiniTools are already bundled client-side in `lib/std/bundle.js` (17MB). Serving them individually through the registry endpoint was deferred. This means the registry endpoint works for capabilities and stdio/http MCPs but not for individual MiniTool resolution. Not blocking, but a gap in the "unified endpoint" vision.

### 4. Story Interdependencies Created Sequencing Pressure

The dependency chain was strictly sequential: 14.1 -> 14.2 -> 14.3 -> 14.3b -> 14.4 -> 14.5 -> 14.6 -> 14.7 -> 14.8 -> 14.11. No parallelization was possible. Each story's completion unblocked the next. This is inherent to the layered architecture but slowed overall velocity.

### 5. Cumulative Test Count Confusion

Test counts are reported both per-story and cumulative across the package. For example, 14.3b reports "199 tests passing" but that includes all prior story tests. Actual new tests per story are harder to extract. Future stories should report *new tests added* separately.

## Key Lessons Learned

### 1. Stdio Mode Constraints Dictate Architecture

When stdin/stdout is the JSON-RPC transport, you cannot use it for user interaction. Any interactive flow (HIL, dependency approval, API key prompts) MUST go through the MCP response/callback pattern. Design for this constraint from day one.

### 2. Per-Project State > Global State

Moving from `~/.pml/` to `${workspace}/.pml/` was a mid-implementation correction (2026-01-09). Per-project isolation means:
- Projects don't share lockfiles (no cross-project integrity leaks)
- Clone includes lockfile (reproducible builds)
- Multiple projects can run simultaneously without conflicts

### 3. Sandbox Capability Code, Not MCP Servers

The initial instinct was to sandbox everything. The refined insight: MCP servers *need* system access (that is their job). Sandbox the *untrusted capability code* from the registry. Protect MCP server calls with HIL approval. This separation of concerns is the right model.

### 4. Hardcoded Mappings Are Tech Debt with an Expiry Date

Story 14.6 hardcodes tool-to-API-key mappings (`TOOL_REQUIRED_KEYS`). The story explicitly marks this as "MVP: later from registry metadata" and implements the dynamic path (`mcpDeps[].envRequired`). This is the right approach: ship the hardcoded version, build the dynamic path, deprecate the hardcoded one. But the deprecation must actually happen.

### 5. Hash-Based Integrity Has Limits

The 4-character hash in FQDNs (65,536 combinations per namespace) is a pragmatic tradeoff for URL readability. The full SHA-256 is stored in the lockfile. If collisions become a problem, extending to 6 characters (16M combinations) is straightforward.

### 6. Public/Private Repo Split Works

The architecture of syncing `packages/pml/` to a public repo and building releases there cleanly separates open source from proprietary code. The sync workflows (`sync-pml-package.yml`, `sync-to-public.yml`) are the glue.

## Patterns & Insights

### 1. MCP Routing Architecture Pattern

The routing architecture established a reusable pattern:

```
Tool Call
  -> Permission Check (user's .pml.json: allow/deny/ask)
  -> Routing Resolution (platform's mcp-routing.json: client/server)
  -> Execution:
       client -> Sandbox Worker (permissions: none) -> mcp.* RPC -> MCP Server
       server -> HTTP forward to cloud -> BYOK key injection -> Response
```

**Reusable insight**: Separate *who controls what* (platform controls routing, user controls permissions) from *where things execute* (client vs server).

### 2. BYOK Pattern

API keys never leave the user's machine for local execution. For cloud execution, keys are injected per-request and never stored server-side. The `.env` reload-on-continue pattern (user edits `.env`, clicks Continue, PML reloads) is a good UX compromise.

### 3. Lockfile-Based Supply Chain Security

The lockfile pattern (first fetch via HTTPS, store hash, verify on subsequent fetches, HIL on hash change) provides supply chain security without requiring a separate signing infrastructure. It is the npm/deno lock pattern applied to MCP distribution.

### 4. Approval Type Extension Pattern

Three approval types emerged organically:
1. `dependency` -- install MCP dependency (14.3b)
2. `api_key_required` -- missing API key (14.6)
3. `integrity` -- hash mismatch in lockfile (14.7)

All follow the same `approval_required` -> `continue_workflow` pattern. New approval types can be added without changing the flow.

## Summary

Epic 14 delivered a functional hybrid local/cloud MCP gateway in 12 days across 9 stories with 731+ tests. The architecture is sound: platform-controlled routing, user-controlled permissions, sandboxed capability execution, and supply-chain integrity via lockfiles.

The two backlog stories (14-9, 14-10) are genuinely lower priority -- the core distribution and execution model works without them. The main technical debt is the hardcoded API key mapping and the unfinished HTTP serve mode.

The biggest architectural win is the separation of concerns: untrusted code is sandboxed, dangerous operations require HIL approval, and integrity is verified via lockfile. This three-layer security model is the right foundation for distributing third-party MCP capabilities.
