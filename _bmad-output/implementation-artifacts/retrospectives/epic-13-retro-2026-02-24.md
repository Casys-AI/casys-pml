## 2. What Went Well

### 2.1 Story Consolidation (15 to 9)

The original epic had 15 fine-grained stories. Consolidation down to 9 reduced overhead without losing coverage:

- Merge stories that touch the same files (e.g., 13.1+13.2+alias parts of 13.7 share `capability-registry.ts`)
- Merge conceptually coupled stories (e.g., "name parameter" + "call by name" are two halves of the same feature)
- Keep stories with independent deployment value separate (e.g., 13.9 Routing Inheritance stands alone)

Saved an estimated 5 dev-days compared to the original 25-day plan.

### 2.2 Clean Dual-Table Architecture

The decision to keep `capability_records` separate from `workflow_pattern` and link via FK (migration 023):
- No risky migration of production data
- Each table has a clear responsibility (registry vs. code/execution)
- The FK link eliminates data duplication from early designs

### 2.3 FQDN Design Proved Robust

The `<org>.<project>.<namespace>.<action>.<hash>` format worked well:
- Immutable identity (code changes = new hash = new FQDN)
- display_name freely mutable without breaking references
- Alias chain prevention (A -> C, not A -> B -> C) simple to implement and correct

### 2.4 Free-Form Naming Over Enforced Namespaces

Allowing free-form `display_name` instead of enforced `namespace:action` was pragmatic. The MCP-compatible validation (`isValidMCPName`) is the only constraint, which is sufficient.

### 2.5 Feature Module Pattern (13-3)

CapabilityMCPServer followed the Feature Module Pattern cleanly: interface-first design, service layer separation, max 3-4 dependencies per class, 31 unit + 3 integration tests.

### 2.6 Smart Skipping and Deferring

- **13-4**: No user demand for auto-naming. Manual `cap:rename` is sufficient.
- **13-6**: Versioning without `cap:update` is meaningless. Code is immutable after creation.

## 3. Challenges & Struggles

### 3.1 Naming Complexity Underestimated

Four different representations of the same entity increase cognitive load:

| Representation | Example                            | Mutable | Used By         |
| -------------- | ---------------------------------- | ------- | --------------- |
| FQDN (PK)     | `local.default.fs.read_json.a7f3`  | No      | Internal refs   |
| display_name   | `json-reader`                      | Yes     | Users           |
| MCP tool name  | `mcp__fs__read_json`               | No      | Claude/clients  |
| Alias          | `old-reader`                       | No      | Backward compat |

Correct but hard to onboard newcomers.

### 3.2 Scope Resolution Deferred

The `Scope { org, project }` mechanism is hardcoded to `{ org: "local", project: "default" }`. Multi-tenant scope resolution was designed in the schema but never implemented.

### 3.3 Accept Suggestion API Churn

Story 13-2 redesigned the execution API mid-implementation. Original: `capability` + `args` top-level params. Final: `accept_suggestion: { callName, args }` response pattern. Correct redesign, but caused rework.

### 3.4 Dual-Table JOIN Complexity

The FK link between `capability_records` and `workflow_pattern` means every code fetch requires a JOIN. Stories 13-2 and 13-3 both had to update `data-service.ts` for this.

## 4. Key Lessons Learned

### 4.1 Merge Stories That Share Files

When two stories modify the same 3+ files, merge them. The 13.1+13.2+13.7 merge was the best example -- avoided merge conflicts and duplicated context loading.

### 4.2 Skip Early, Skip Explicitly

Story 13-4 was skipped during planning, not after implementation started. Deferral reason documented directly in the story file.

### 4.3 Schema-First, Code-Second

Starting with migration 021 (full schema) before writing application code meant the data model was stable from day one. No "add column" migrations needed for the core schema.

### 4.4 Immutable PKs Prevent Cascading Renames

FQDN as immutable PK was the most important design decision. If display_name were the PK, every rename would cascade through FKs, aliases, saved code references, and tool_usage records.

### 4.5 Config Over Code for Runtime Lists

Story 13-9 used `config/mcp-routing.json` instead of hardcoded lists. Server routing can change without code changes.

## 6. Patterns & Insights

### 6.1 Convention Over Enforcement

Standard namespaces (`fs:`, `api:`, `db:`) are recommended, not required. Avoids the classification problem without losing structure.

### 6.2 FQDN as Internal Plumbing

FQDNs are hidden from agents (FR034). Agents see display_name or `mcp__ns__action`. FQDN format can evolve without breaking agent interactions.

### 6.3 Capability = Transparent MCP Tool

Capabilities exposed as `mcp__<namespace>__<action>` are indistinguishable from native MCP tools. This transparency is the entire point.

### 6.4 Single-File Module Pattern

`lib/std/cap.ts` (~440 lines) instead of `lib/std/mcp/` directory. Don't create directory structure prematurely.

### 6.5 Routing: Local-First Security

Unknown servers default to `local`. Only explicitly listed cloud servers get `routing = "cloud"`. Correct security posture.

## 7. Recommendations for Future Work

1. **cap:update + Versioning (13-6)**: Implement `cap:update` with dry-mode validation, then unblock versioning, `capability_versions` table, version specifiers.

2. **Scope Resolution for Multi-Org**: Extract `Scope` from session/auth context, implement resolution order (exact scope -> org-wide -> public), add visibility enforcement.

3. **Evaluate Table Consolidation**: Consider merging `capability_records` into `workflow_pattern` as optional columns to reduce JOIN complexity. Weigh against separation of concerns.

4. **Capability Output Schema**: Add return-type inference to execution pipeline for richer DAG analysis and MCP tool introspection.

5. **Story Sizing Guideline**: For future epics, apply consolidation upfront:
   - **Merge threshold**: 2 stories sharing 3+ modified files
   - **Skip threshold**: story's only consumer is another unplanned story
   - **Defer threshold**: prerequisites don't exist, document blockers explicitly
