## 2. What Went Well

**The spike was the right call.** `spike-search-tools-graph-traversal.md` correctly identified PageRank as the wrong algorithm for search, proposed Adamic-Adar, and laid out the Netflix-style pipeline. The stories executed cleanly because the research was done upfront. No surprises during implementation.

**ADR-022 prevented code duplication.** Story 5.2 extracted `searchToolsHybrid()` into the engine before adding templates, so `gateway-server.ts` (search_tools handler) and `dag-suggester.ts` (suggestDAG) both use the same scoring path. Good architectural discipline.

**Adaptive alpha is genuinely smart.** The alpha parameter auto-adjusts based on graph density (0 edges = pure semantic, 200+ edges = 50/50 hybrid). This means the system works on day one with zero data, then gracefully transitions to graph-augmented search as usage data accumulates. No manual tuning needed.

**Test coverage is excellent for Story 5.2.** 52 tests across loader validation, edge conversion, DB sync, checksum comparison, bootstrap scenarios, and hybrid search. The code review gave 5-star ratings across all quality categories.

**Performance is a non-issue.** `searchToolsHybrid` overhead measured at 0.4ms. Graph sync at 1.2ms. Well below any reasonable threshold.

## 3. Challenges & Struggles

**Honestly, this was a clean epic.** Two stories, clear spike research, no blockers, no architectural surprises. The main friction points were minor:

- **AC2 deviation (Story 5.1):** Default limit was implemented as 5 instead of the specified 10. Caught in review, flagged as low-priority action item. Never fixed. This is the kind of thing that festers -- small, "non-blocking," and easy to forget.

- **No integration test for `handleSearchTools`**: The review noted this gap. Unit tests cover the engine methods, but the gateway handler itself has no direct test. The handler is ~100 lines of glue code that could drift.

- **Adamic-Adar not in Graphology:** Required custom implementation. Not hard, but it means we own the algorithm code and need to trust our implementation is correct. No peer-reviewed library backing it.

---

## 4. Key Lessons Learned

1. **Spikes pay for themselves in small epics.** This epic took ~4-6h for 2 stories. Without the spike, we'd have spent hours debating PageRank vs Adamic-Adar during implementation. The spike compressed that into a clean document and an obvious choice.

2. **Adaptive parameters > hardcoded thresholds.** The original `execute_workflow` had a hardcoded 0.50 confidence threshold that blocked valid matches (0.48 for "screenshot"). The adaptive alpha in Epic 5 eliminates this class of problem. Design lesson: when a threshold exists, ask "should this adapt to context?"

3. **Separation of concerns clarifies everything.** The old `execute_workflow` mixed search + DAG construction. Splitting into `search_tools` (find tools) and `execute_workflow` (build/run DAG) made each simpler, more testable, and independently improvable.

4. **YAML templates for cold start is pragmatic.** The bootstrap mechanism (if 0 edges, auto-sync templates) solves the chicken-and-egg problem without ML or complex initialization. Sometimes the best solution is the simplest one.

## 6. Patterns & Insights

**Pattern: Spike -> ADR -> Stories works well for algorithm-heavy epics.** The flow was: spike identified the right algorithm (Adamic-Adar), ADR-022 established the hybrid search architecture, then stories implemented it cleanly. This pattern should be repeated for any epic involving algorithm selection.

**Pattern: Bootstrap-then-learn.** Templates provide initial graph edges, which get augmented by real usage data over time. This pattern recurred later (e.g., SHGAT training with synthetic data, GRU with workflow templates). It's a core PML pattern: never start from zero, always have a baseline.

**Insight: Graph algorithms need production data to prove value.** At the time of Epic 5, the graph had very few edges. The adaptive alpha compensated (high alpha = mostly semantic), but the real graph-based re-ranking power only materializes with enough usage history. The 0.4ms overhead is fine, but the re-ranking benefit was negligible at launch. This is acceptable -- the infrastructure was correctly laid for future value.

**Insight: This epic aged well.** The `searchToolsHybrid()` method became the foundation for `pml_discover` (Story 10.6), SHGAT scoring integration (Story 10.7b), and capability matching (Story 7.3a). The ADR-022 decision to centralize hybrid search was validated by at least 3 downstream consumers.

## 7. Recommendations for Future Work

1. **Close the AC2 gap.** Change default limit from 5 to 10, or explicitly update the spec to say 5. Don't leave specs and code contradicting each other.

2. **Add an integration test for the search_tools handler.** The handler assembles semantic results + graph re-ranking + meta response. A single integration test hitting the handler directly (not just the engine) would catch regressions in the glue code.

3. **Monitor graph re-ranking impact.** Now that the system has months of production data, measure how often graph re-ranking actually changes the top-K ordering versus pure semantic. If it rarely changes anything, the complexity may not be justified. If it significantly improves results, document it as a validated win.

4. **Consider replacing Adamic-Adar with SHGAT scores.** The SHGAT model (Epics 10-11) now provides learned tool relatedness that's likely superior to the structural Adamic-Adar metric. Evaluate whether `searchToolsHybrid()` should use SHGAT scores as the graph component instead of (or in addition to) Adamic-Adar.

5. **Keep the spike-first discipline for algorithm choices.** This epic proved it works. Don't skip it for future graph/ML epics.
