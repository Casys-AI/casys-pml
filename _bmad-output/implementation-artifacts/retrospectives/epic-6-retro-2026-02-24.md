## 2. What Went Well

### Incremental, Layered Architecture
Stories 6.1-6.4 built cleanly on top of each other. SSE infrastructure (6.1) was reused by all subsequent stories without modification. The EventsStreamManager was created once and consumed everywhere.

### Consistently High Quality Scores
Three stories scored 95-100/100 in code review. Zero HIGH or MEDIUM severity issues across the entire epic.

### Test Volume and Coverage
115 tests across 6 stories. Stories 6.3 (23 tests) and 6.5 (43 tests) stand out. Performance benchmarks included where relevant (<1ms event emission, <10ms search latency, <500ms graph render).

### EventBus Unification (6.5) Was the Right Call
Before 6.5, event emission was scattered across EventTarget, BroadcastChannel, internal arrays, and direct SSE coupling. The ADR-036-driven unification into a single EventBus singleton simplified the SSE handler significantly.

### UX Design Review Paid Off (6.4)
The UX Designer agent review on 6.4 caught real problems from a dashboard screenshot -- all nodes were grey ("unknown" server), labels were illegible. This led to a bug fix and the Neo4j "Color on Demand" pattern.

### CDN Strategy for Frontend Libraries
Using Cytoscape.js and Chart.js via CDN avoided build pipeline complexity. Pragmatic choice for a dashboard that ships as static HTML/islands.

## 3. Challenges & Struggles

### Story 6.2: Documentation Task Left Incomplete
Task 10 in Story 6.2 was left undone. Creates a pattern where documentation is the first thing to be cut.

### Story 6.3: Missing Database Tables (Epic 2 Debt)
The `workflow_execution` table defined in Epic 2's SQL migration file was never integrated into the TypeScript migration system. This caused `getMetricsTimeSeries()` and `getPeriodStats()` to fail silently at runtime. Migration 010 was created to fix it.

### Story 6.4: Server Mapping Bug in Graph Snapshot
`getGraphSnapshot()` used `toolId.split("__")[1]` but real tool IDs use `server:tool` format (colon separator, not double underscore). Every node showed `server = "unknown"`. Required a Task 0 bug fix.

### Story 6.5: Cross-Epic Dependency
Story 6.5 was blocked by Story 7.3b (capability injection with BroadcastChannel tracing). This cross-epic dependency broke the flow of Epic 6 completion.

### Hard-coded API Base URL in Fresh Islands
Multiple stories noted that `apiBase` URL is hard-coded to `http://localhost:3001` in islands due to a Fresh props serialization limitation. Will break in any non-localhost deployment.

## 4. Key Lessons Learned

1. **Get the event format right in Story 1.** The SSE event format and GraphEvent type union established in 6.1 were reused unchanged through 6.5. Investing time in foundation event types paid dividends.

2. **Check data format assumptions with real data.** The `split("__")` vs `split(":")` bug would have been caught instantly by testing with real tool IDs. Always verify format assumptions against actual production data.

3. **SQL migration files must be wired into the runner.** A `.sql` file in the migrations directory is not a migration unless the TypeScript migration system knows about it. Treat unwired migration files as bugs.

4. **UX review with real screenshots catches what code review cannot.** The 6.4 UX review identified grey nodes and illegible labels -- things that pass all unit tests but are immediately visible to a human.

5. **Cross-epic dependencies should be explicit in sprint planning.** Story 6.5 depending on 7.3b was documented but still caused a gap.

6. **Dual update strategy (SSE + polling) is resilient.** Story 6.3's approach of polling every 5s with SSE as a fast path handles SSE disconnects gracefully.

## 6. Patterns & Insights

### Pattern: SSE as the Universal Dashboard Transport
Every frontend-facing story uses SSE. The pattern is consistent: backend emits events, SSE endpoint streams them, browser `EventSource` consumes them. Simpler than WebSockets for unidirectional data flow.

### Pattern: Fresh Islands for Interactive Panels
`GraphVisualization.tsx`, `MetricsPanel.tsx`, `GraphExplorer.tsx`, and `AdminDashboardIsland.tsx` are all islands with `useState`/`useEffect` for local state. Static components are pure prop-driven.

### Pattern: GraphRAGEngine as the Single Data Source
All API endpoints delegate to methods on `GraphRAGEngine`. No story introduced a competing data source.

### Pattern: EventBus Singleton + BroadcastChannel
Story 6.5 established the canonical event distribution pattern. Consumers subscribe via `eventBus.on(type | "*", handler)`. Replaced 4 prior mechanisms.

### Anti-Pattern: Assumptions About ID Formats
The `split("__")` bug is a recurring theme. Tool IDs exist in at least 3 formats (`server:tool`, `mcp__server__tool`, `org.project.namespace.action.hash`). Any code that parses tool IDs must handle all formats or use a centralized normalizer.

### Insight: Quality Scores Correlate With Foundation Reuse
Story 6.1 (100/100) established the foundation. Stories reusing proven infrastructure tend to score higher.

## 7. Recommendations for Future Work

1. **Fix the hard-coded `apiBase` in islands.** Most impactful debt item. Investigate Fresh's `ctx.state` or environment variable injection at build time.

2. **Add ambient type declarations for CDN libraries.** Create `types/chart.js.d.ts` and `types/cytoscape.d.ts` to eliminate `@ts-ignore`.

3. **Add E2E browser tests for the dashboard.** Playwright or similar test suite verifying SSE real-time updates.

4. **Audit and fix the 19 pre-existing test failures.** Broken tests erode confidence in the test suite overall.

5. **Consider CDN fallback or bundling.** For offline or air-gapped environments.

6. **EventBus event type documentation.** Generated reference from the TypeScript union.

---

**Verdict:** Solid observability epic. 115 tests, consistently high quality scores. EventBus unification (6.5) was the most impactful architectural decision. Main debt is hard-coded `apiBase` and CDN dependencies.
