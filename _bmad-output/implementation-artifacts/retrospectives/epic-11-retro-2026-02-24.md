## What Went Well

### 1. Clean RL Architecture (DQN/Rainbow Pattern)

The TD Error + PER + SHGAT architecture is conceptually sound. Each component has a clear, single role:

- **TD Error** = learning signal (`actual - predicted`)
- **PER** = replay prioritization (`priority = |td_error|`)
- **SHGAT** = the model itself (attention weights, path prediction)

This separation made each story testable in isolation. The pipeline flows naturally: execution -> trace -> TD error -> priority -> PER sample -> train SHGAT -> update priorities. No circular dependencies.

### 2. Aggressive Deprecation Was the Right Call

Removing `CapabilityLearning` entirely (rather than keeping it alongside SHGAT) simplified 11.3 dramatically. The original plan had 11.3 producing intermediate stats that 11.6 would consume. The refactored version (2025-12-22) cut that to: 11.3 produces `priority`, 11.6 samples by priority. Less code, fewer seams, same outcome.

### 3. Schema Cleanup First (11.0) Paid Off

Starting with DB cleanup before any learning work removed friction for every subsequent story. Migration 019 eliminated 3 redundant tables and fixed FKs. The KV singleton (`src/cache/kv.ts`) became a reusable pattern beyond this epic.

### 4. Decisions[] Inference Was a Good Surprise

Story 11.2 originally required runtime branch decision tracking. The AI review flagged this as unnecessary: `StaticStructureBuilder.inferDecisions()` matches executed tools against conditional edges post-hoc. No runtime instrumentation needed. 8 tests added. This is a pattern worth reusing -- infer from static structure + trace rather than instrumenting at runtime.

### 5. Velocity: 3-Day Core Delivery

Stories 11.0 through 11.6 (excluding 11.4 UI) were implemented in 3 days (Dec 22-24). The sequential dependency chain (11.0 -> 11.1 -> 11.2 -> 11.3 -> 11.6) was handled without blocking because each story was scoped to 0.5-2 days.

## Challenges and Struggles

### 1. API Mismatch Between Spec and Implementation (11.3)

The original AC for 11.3 specified `calculateTDError(shgat, trace)` but the implementation required an `EmbeddingProvider` parameter (for intent embeddings used in multi-head scoring). Same issue with `predictPathSuccess(path)` vs `predictPathSuccess(intentEmbedding, path)`.

**Root cause:** The spec was written before understanding that SHGAT's multi-head architecture requires intent embeddings for scoring. The spec assumed a simpler path-only API.

**Lesson:** When the spec says "predict from path", verify whether the underlying model needs additional inputs. Update ACs to match the actual interface before marking done.

### 2. contextTools Was a Dead Field (11.6)

Critical discovery during 11.6: `contextTools` in `TrainingExample` was passed but never referenced in `trainBatch()`. The system was predicting on paths (`predictPathSuccess`) but not learning from paths. This meant SHGAT was doing tool-level learning disguised as path-level.

**Fix:** Multi-example generation per trace (N nodes -> N training examples) where each example uses `contextTools = path.slice(0, i)`.

**Lesson:** When a model has separate predict and train paths, verify they use the same features. A dead field in training is a silent accuracy leak.

### 3. PER Sampling Implementation Subtlety

The initial SQL-based `sampleByPriority()` from 11.2 used `ORDER BY priority * random() DESC` -- not true PER sampling (P(i) proportional to priority^alpha). Had to implement correct Schaul et al. 2015 algorithm in TypeScript with reweighting after each selection.

**Lesson:** Approximate sampling in SQL is fine for display but not for RL training. When the algorithm matters, implement it in application code.

### 4. Null Handling Bug (11.1)

Code review caught a null handling bug in `worker-bridge.ts` -- optional chaining was needed for `mcpResult?.isError`. Small fix, but it would have caused runtime crashes on tools returning null results.

**Lesson:** Result tracing touches the hot path of every tool execution. Defensive coding is mandatory here.

### 5. Story 11.4 Scope Creep

Originally a simple toggle UI, 11.4 expanded to include: `layerIndex` for fan-in/fan-out edges, fixing `execution-learning.ts` parallel edge generation, adding `provides`/`dependsOn` definition edges, and the split CodePanel layout. The story went from an estimated 2 days to a feature with 13 ACs across backend, API, and UI. It remains the only undelivered core story.

## Key Lessons Learned

### RL System Design

1. **Priority = |TD error| is elegant but needs cold start handling.** Default 0.5 works, but all traces get the same priority until SHGAT has enough data. The system is effectively doing uniform sampling during cold start, which is correct behavior but should be documented.

2. **Multi-example per trace > single-example per trace.** A path of 3 nodes generates 3 training examples with progressively longer context. This is closer to how sequence models learn and gives more gradient signal per trace.

3. **Weighted average with later-nodes-higher is a reasonable path score.** Weights `1.0, 1.5, 2.0, 2.5...` give more importance to later nodes in the path (the "hard" steps). Simple, interpretable, no hyperparameter tuning needed.

4. **Training lock prevents concurrent corruption.** `isTrainingInProgress` flag with early return is minimal but effective. No mutex needed since Deno is single-threaded for JS.

### Architecture

5. **Capability vs Trace separation is fundamental.** Capability = what CAN happen (static, immutable). Trace = what DID happen (dynamic, per-execution). Mixing these (as the old `saveCapability` did) creates confusion about lifecycle and mutability.

6. **Unit of Work pattern for capability+trace creation.** `saveCapability()` returns `{ capability, trace }` and creates both atomically when `traceData` is provided. This ensures FK integrity without requiring two-phase commit.

7. **Sanitization must happen at the storage boundary.** `sanitizeForStorage()` redacts sensitive data and truncates payloads >10KB. Doing this in the store layer (not the caller) ensures no leak path.

## Patterns and Insights

### DQN/Rainbow Pattern Applied to MCP Tool Selection

The architecture maps directly from RL game-playing to tool selection:

| RL Concept | AgentCards Implementation |
| ---------- | ------------------------ |
| State | Intent embedding + context tools |
| Action | Tool/capability selection |
| Reward | Execution success (binary) |
| Experience replay | `execution_trace` table |
| Prioritized replay | `priority = \|td_error\|` |
| Q-network | SHGAT multi-head attention |
| Target network | Not implemented (single network) |

**Missing from full Rainbow:** No target network (would reduce training instability), no n-step returns (traces are single-step success/fail), no distributional value (binary outcome only). These are valid simplifications for the current data volume.

### The "Infer, Don't Instrument" Pattern

Story 11.2's `inferDecisions()` demonstrated that runtime instrumentation can often be replaced by post-hoc inference from static structure + trace data. This is cheaper, less invasive, and easier to test. The tradeoff is accuracy: inference can miss decisions that don't leave tool-level traces.

### Additive Migration Strategy

All migrations used `IF EXISTS` / `IF NOT EXISTS` guards. Migration 019 deletes tables created by migrations 004, 008, 009 without modifying those original migrations. The migration chain remains replayable from scratch.

## Recommendations for Future Work

### 1. Implement 11.4 as a Focused UI Story

Strip 11.4 to its core: the split CodePanel with real trace data from `ExecutionTraceStore`. Defer `layerIndex`, fan-in/fan-out edges, and `provides`/`dependsOn` edges to separate stories. The current 13-AC scope is why it stalled.

### 2. Monitor PER Priority Distribution

Add a metrics endpoint or log that shows the distribution of priorities across traces. If all priorities cluster around 0.5 (cold start never ends) or 0.0 (SHGAT overconfident), the learning signal is weak. This is the canary for the RL system.

### 3. Consider Target Network for Training Stability

Currently, `predictPathSuccess()` and `trainBatch()` use the same SHGAT instance. After training, priorities are recalculated with the updated model, which can cause oscillation. A frozen target network (updated every N training cycles) would stabilize priority estimates.

### 4. Evaluate Path Flattening vs Hierarchical Learning

`flattenExecutedPath()` currently inserts child traces sequentially. With the capability hierarchy growing (L0 tools, L1 caps, L2 meta-caps), hierarchical path encoding might outperform flat sequences. This connects to the GRU work (see `memory/gru-training.md`).

### 5. Close the Data Quality Loop

The `executed_path` field has known corruption issues (18.4% UUID, 6.1% FQDN as of Feb 2026). `task_results` is the clean source. Ensure all training pipelines use `task_results` as primary and `executed_path` only as fallback. This was partially fixed in Feb 2026 but should be verified end-to-end.

## Summary

Epic 11 delivered the foundational learning infrastructure: trace capture, persistence, TD error signal, PER prioritization, and SHGAT path-level training. The DQN/Rainbow-inspired architecture is clean and each component is independently testable. The main gap is the UI story (11.4) which needs scope reduction before implementation. The biggest risk going forward is not code quality but data quality -- the learning system is only as good as the traces it trains on.
