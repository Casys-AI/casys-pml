## Overview

### Problem Statement

Le client PML lit la config MCP depuis `.pml.json` (command, args, env) mais **ne l'envoie pas** au serveur lors du sync. Résultat : le serveur ne peut pas retourner les infos de spawn, et `pml_execute` échoue avec "Stdio capability missing install info".

### Solution

Utiliser la table `tool_observations` existante (déjà multi-tenant par user) et transformer la colonne `observed_args TEXT[]` en `observed_config JSONB` pour stocker la config complète.

### Flux actuel (cassé)

```

──────────────────                      ────────────
```

### Flux corrigé

```
CLIENT (.pml.json)                      SERVEUR (DB)
──────────────────                      ────────────
mcpServers: {                           tool_schema: ✅ tools

    command: "deno",                      observed_config: {  ← NOUVEAU
    args: [...],                            command: "deno",
    env: {...}                              args: [...],
  }                                         env: {...}
}                                         }
     ↓
tool-sync.ts envoie {                   → deriveEnvRequired(config) ✅
  tools,                                → metadata.install = { command, args } ✅
  observedConfig ← NOUVEAU              → spawn OK ✅
}
```

#### Task 2b: Fonction sanitizeEnvToPlaceholders

```typescript
/**
 * Convert env values to placeholders for safe transmission.
 * NEVER send actual API keys to the server.
 *
 * @example { "API_KEY": "sk-secret" } → { "API_KEY": "${API_KEY}" }
 */
function sanitizeEnvToPlaceholders(
  env: Record<string, string> | undefined
): Record<string, string> | undefined {
  if (!env) return undefined;
  const sanitized: Record<string, string> = {};
  for (const key of Object.keys(env)) {
    sanitized[key] = `\${${key}}`;
  }
  return sanitized;
}
```

## Security

### Env Vars - Règle stricte

```typescript
// ❌ INTERDIT - expose les secrets
{ env: { "API_KEY": "sk-actual-secret-value" } }

// ✅ REQUIS - seulement les placeholders
{ env: { "API_KEY": "${API_KEY}" } }
```

La fonction `sanitizeEnvToPlaceholders()` garantit que seuls les noms des variables sont transmis. Les vraies valeurs restent côté client et sont résolues au moment du spawn.

## Notes

### Pourquoi `tool_observations` et pas une nouvelle table ?

1. **Existe déjà** avec la bonne structure multi-tenant (user_id)
2. **Permet différentes configs par user** pour le même serveur
3. **Évite de créer une nouvelle table** et les jointures associées

### Breaking change `observed_args` → `observed_config`

- `observed_args TEXT[]` n'était jamais peuplé (toujours `{}`)
- Pas de données à migrer
- La contrainte d'unicité change : `(user_id, tool_id, observed_args)` → `(user_id, server_namespace)`

### `deriveEnvRequired()` fonctionne automatiquement

La fonction existe déjà dans `src/mcp/registry/hash-utils.ts:91`. Elle extrait les noms des env vars depuis `config.env` où la valeur commence par `${`. Une fois la config stockée en DB, cette fonction retournera les bonnes valeurs sans modification.

### Compatibilité PGlite

Le `COMMENT ON COLUMN` dans la migration est wrappé dans un try-catch car PGlite peut ne pas supporter cette syntaxe. La migration fonctionne sur les deux backends (PostgreSQL et PGlite).
