# 05.5 - X11/WebKit Issues (Remote Development)

## Context

During testing of Increment 5 (Terminal Integration) via X11 forwarding from a remote Linux server to Windows (MobaXterm), several issues were encountered.

**Commit terminal implementation:** `2077a64b`

## Issues

### 1. Black Screen with Default WebKit Rendering

**Symptom:** Tauri window appears but content is completely black.

**Cause:** WebKit uses GPU compositing by default. X11 forwarding cannot forward GPU-rendered frames.

**Workarounds testés:**
```bash
# Celui qui a marché:
WEBKIT_DISABLE_COMPOSITING_MODE=1 cargo tauri dev

# Autre option (non testée):
LIBGL_ALWAYS_SOFTWARE=1 cargo tauri dev
```

### 2. "Load failed" API Calls

**Symptom:** App loads but API calls to `localhost:3003` fail with "Load failed".

**Attempted fixes:**
- Updated CSP to allow `connect-src http://localhost:3003`
- Disabled CSP entirely (`"csp": null`)

**Status:** Unresolved. Suspected cause: WebKit in software rendering mode may have network request limitations.

**Next steps to investigate:**
1. Test on local machine (not via X11) to confirm it's X11-specific
2. Check WebKit logs for detailed error
3. Try alternative: `WEBKIT_DISABLE_DMABUF_RENDERER=1`
4. Consider using WebKitGTK inspector for debugging

## VcXsrv Settings Testés

- Multiple windows ✓
- Display number: 0
- Start no client ✓
- Disable access control ✓ (important!)
- Native OpenGL: décoché

xclock fonctionnait via MobaXterm, confirmant que X11 forwarding de base marchait.

## X11 Forwarding Setup (MobaXterm)

For future reference, working X11 setup:

1. Install MobaXterm on Windows
2. Create SSH session with private key
3. MobaXterm auto-starts X server
4. Use `WEBKIT_DISABLE_COMPOSITING_MODE=1` for Tauri

**Note:** Windows OpenSSH supporte X11 forwarding depuis v8.1p1 (Windows 1809+), mais nécessite:
1. `$env:DISPLAY="127.0.0.1:0.0"` sur Windows AVANT connexion
2. `ssh -Y` (pas juste `-X`)
3. VcXsrv avec "Disable access control" coché

Dans notre cas, malgré DISPLAY=localhost:10.0 côté serveur, les fenêtres n'apparaissaient pas sur VcXsrv (0 clients connectés). Possiblement une étape de config manquée. MobaXterm a fonctionné immédiatement sans config supplémentaire.

## Autres problèmes rencontrés

### Port 1420 déjà utilisé
Vite dev server utilise le port 1420. En cas de conflit:
```bash
fuser -k 1420/tcp
```

### CSP actuel (désactivé pour debug)
Le fichier `tauri.conf.json` a actuellement `"csp": null`. À restaurer une fois le problème résolu:
```json
"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src 'self' http://localhost:3003"
```

## Recommendation

For Tauri development requiring visual testing:
- **Preferred:** Develop on local machine with display
- **Alternative:** Use VNC/RDP instead of X11 forwarding
- **Workaround:** X11 + software rendering (limited functionality)
